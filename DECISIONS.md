# ADR-001 使用 Vercel 作为唯一后端平台

**状态**

✅ Accepted

## 决策

整个项目统一使用 Vercel 提供：

- Hosting
- Functions
- KV
- Deployment

不引入第二个后端平台。

## 原因

本项目由个人长期维护。

减少平台数量可以显著降低：

- 运维成本
- 学习成本
- 部署复杂度
- 故障排查难度

相比追求最佳技术方案，更优先降低维护成本。

## 放弃方案

- Supabase
- Firebase
- Railway
- Render
- 自建服务器

# ADR-002 第一阶段采用 H5，而不是微信小程序

**状态**

✅ Accepted

## 决策

第一阶段仅开发 H5。

微信小程序作为后续新增客户端。

## 原因

当前主要用户：

印尼工业园区中国用户。

H5：

- 分享方便
- 无需审核
- 发布快
- 更新快

能够快速验证产品价值。

## 放弃方案

第一阶段直接开发微信小程序。

原因：

增加开发成本。

增加审核流程。

降低开发效率。

# ADR-003 微信小程序仅作为客户端

**状态**

✅ Accepted

## 决策

未来增加微信小程序时：

仅作为 UI 客户端。

业务逻辑仍然运行于：

Vercel。

## 原因

微信云开发服务器主要位于国内。

主要用户位于印尼。

网络延迟较高。

保持：

一套后端。

多个客户端。

避免维护两套系统。

# ADR-004 使用 Event Sourcing

**状态**

✅ Accepted

## 决策

采用 Event Sourcing。

Event Log 为唯一事实来源。

当前状态通过 Replay Event 得到。

## 原因

天然支持：

- Undo
- Replay
- 历史记录
- 调试
- 数据恢复

业务逻辑更加简单。

## 放弃方案

直接保存当前分数。

原因：

历史难以恢复。

撤销复杂。

容易出现数据不一致。

# ADR-005 使用 Redis List 保存 Event

**状态**

✅ Accepted

## 决策

Room：

Redis Hash

Event：

Redis List

新增：

RPUSH

读取：

LRANGE

## 原因

Redis List 天然适合：

Append Only。

与 Event Sourcing 完全契合。

无需整体覆盖 JSON。

维护简单。

## 放弃方案

一个 Room 保存整个 JSON。

原因：

Event 越来越大。

每次修改都需要整体写回。

效率较低。

# ADR-006 使用 Polling，而不是 WebSocket

**状态**

✅ Accepted

## 决策

客户端采用轮询同步。

不使用 WebSocket。

## 原因

麻将计分：

实时性要求较低。

Polling：

实现简单。

维护简单。

更符合 Serverless。

能够满足实际需求。

## 放弃方案

WebSocket。

原因：

增加部署复杂度。

增加维护成本。

收益有限。

# ADR-008 保持 MVP

**状态**

✅ Accepted

## 决策

最核心计分能力。

暂不开发：

- 登录
- 用户体系
- 排行榜

- 广告
- 社交功能

遵循：

YAGNI。

# ADR-009 优先降低维护成本

**状态**

✅ Accepted

## 决策

技术选型优先级：

维护成本

↓

简单

↓

可读性

↓

开发效率

↓

性能

## 原因

本项目属于：

长期维护。

个人开发。

未来维护成本远高于开发成本。

所有新增技术，都应优先考虑：

未来三年是否仍然值得维护。

# ADR-010 AI 开发原则

**状态**

✅ Accepted

## 决策

AI 不得直接开始开发。

必须：

理解需求

↓

分析现有代码

↓

输出方案

↓

等待确认（复杂需求）

↓

开始开发

↓

自检

## 原因

降低返工。

保证架构一致性。

避免 AI 擅自修改项目方向。

# 未来新增 ADR

当出现以下情况时，应新增 ADR：

- 引入新的基础设施
- 更换技术栈
- 调整系统架构
- 修改数据模型
- 调整同步机制
- 放弃既有设计原则

每一项重大技术决策，都应留下记录。

因为：

> **代码会告诉别人系统是如何工作的。**

> **ADR 会告诉别人为什么这样工作。**
