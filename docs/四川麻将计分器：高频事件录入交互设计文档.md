# 四川麻将计分器：高频事件录入交互设计文档

## 1. 文档目标

本文档用于指导 Codex 重构“游戏中事件录入”模块。

本次设计重点不是增加动画、卡片或视觉装饰，而是降低高频录入成本，使用户在牌桌上能够：

- 单手操作；
- 快速识别玩家；
- 尽量减少点击次数；
- 尽量减少手指移动距离；
- 避免 Bottom Sheet 连续切换造成的视觉搜索和手指跳动；
- 在重复使用后形成稳定的肌肉记忆；
- 在保证录入效率的同时，避免无效参数和错误状态。

产品定位：

> 这是一个高频事件录入工具，不是数据展示后台，也不是逐步向导。

核心设计原则：

> 固定布局、渐进启用、最后一步提交、提交后可撤销。

---

# 2. 当前问题

旧版录入流程采用逐步引导：

```text
选择事件
→ 选择玩家
→ 选择另一位玩家
→ 选择番数
→ 确认
```

这种方式虽然逻辑清晰，但存在以下问题：

1. 每一步更新 Bottom Sheet 或页面内容；
2. 按钮位置不断变化；
3. 用户需要反复寻找下一步操作；
4. 手指移动距离增加；
5. 上下文切换次数增加；
6. 高频事件点击次数过多；
7. 无法形成稳定的肌肉记忆；
8. 所有事件被强制套用相同流程，导致简单事件也很慢。

因此，新的录入模块需要改为固定工作台模式。

---

# 3. 核心交互模型

## 3.1 固定工作台

游戏中页面的主要录入区域固定分为四部分：

```text
玩家区
事件区
参数区
反馈区
```

这些区域在整个录入过程中保持相同位置，不因为状态变化而重新排列。

允许变化的只有：

- 高亮状态；
- 禁用状态；
- 提示文字；
- 参数区的可用状态；
- 当前操作语义。

禁止：

- 点击一步后整个页面替换；
- 连续打开多个 Bottom Sheet；
- 玩家按钮重新排序；
- 事件按钮移动；
- 参数区显示和隐藏导致页面整体跳动；
- 每个事件都增加确认弹窗。

---

## 3.2 标准顺序

统一采用：

```text
玩家
→ 事件
→ 必要参数
→ 自动提交
```

但不同事件的必要参数不同。

不能强制所有事件都经过同样数量的步骤。

---

# 4. 页面结构

## 4.1 玩家区

固定展示最多四名玩家。

推荐结构：

```text
[动物头像]
玩家昵称
当前总分
```

示例：

```text
🐼
raisin
+5
```

### 玩家头像规则

- 从内置动物头像中选择；
- 每位玩家头像必须唯一；
- 头像用于快速识别；
- 昵称仍然保留；
- 不建议完全使用“熊猫、狐狸”等动物名称代替真实昵称；
- 昵称可缩小显示，但不能删除；
- 玩家总分可作为辅助信息显示。

### 玩家选中状态

选中玩家后，必须同时改变：

- 边框；
- 背景；
- 头像外圈；
- 昵称字重；
- 顶部提示文本。

不能只依赖轻微的颜色变化。

### 提交后处理

每次事件成功提交后：

```text
立即清除玩家选择
立即清除事件选择
立即清除临时参数
恢复初始状态
```

禁止保留上一位玩家作为默认选择，以免连续误记。

---

## 4.2 事件区

固定展示六个事件：

```text
直杠
补杠
暗杠
点炮
自摸
流局
```

建议保持为 3 × 2 网格：

```text
[直杠] [补杠] [暗杠]
[点炮] [自摸] [流局]
```

### 不合并杠牌

不再将：

- 直杠；
- 补杠；
- 暗杠；

合并为“杠牌”。

原因：

- 这是熟练用户的高频工具；
- 直接展示具体事件能减少一次选择；
- 固定位置有利于形成肌肉记忆；
- 比降低认知负担更重要的是降低长期录入成本。

---

## 4.3 番数区

番数区固定占位：

```text
[1番] [2番] [3番] [4番]
```

不能因为当前事件不需要番数就隐藏该区域，否则会导致页面布局跳动。

### 初始状态

番数区始终可见，但默认禁用。

示例：

```text
番数
[1番] [2番] [3番] [4番]

当前事件无需选择番数
```

### 启用条件

只有系统已经明确事件类型需要番数时，番数区才启用。

允许启用的事件：

- 自摸；
- 点炮。

不允许在“只选择玩家”后就启用番数。

原因：

> 只选择玩家时，系统无法判断下一步是自摸、点炮、直杠、补杠还是暗杠。

因此禁止以下流程：

```text
玩家
→ 3番
→ 自摸
```

因为这会迫使番数在事件未确定时提前启用，并产生无效状态。

正确流程：

```text
玩家
→ 自摸
→ 3番
```

或者：

```text
胡牌者
→ 点炮
→ 点炮者
→ 3番
```

---

## 4.4 反馈区

成功提交后，在固定位置显示轻量反馈：

```text
已记录：raisin 自摸 · 1番
[撤销]
```

反馈可以通过 Toast 或页面底部固定反馈条呈现。

要求：

- 不阻塞后续录入；
- 不弹出确认弹窗；
- 不跳转页面；
- 提供“撤销上一条”；
- 短时间后自动消失；
- 最近一条事件仍可以在主页面被快速撤销。

---

# 5. 状态机设计

建议将录入模块实现为显式状态机，而不是多个布尔值拼接。

建议状态如下：

```ts
type EntryState =
  | { type: "idle" }
  | { type: "player_selected"; actorId: string }
  | { type: "waiting_for_fan"; actorId: string; eventType: "zimo" }
  | {
      type: "waiting_for_counterparty";
      actorId: string;
      eventType: "dianpao" | "zhigang";
    }
  | {
      type: "waiting_for_fan_after_counterparty";
      actorId: string;
      counterpartyId: string;
      eventType: "dianpao";
    }
  | { type: "liuju_mode" };
```

不要只使用：

```ts
selectedPlayer
selectedEvent
selectedFan
```

这种松散状态组合。

原因：

- 容易产生不合法组合；
- 容易让番数在错误时机启用；
- 容易残留上一笔状态；
- 难以判断当前按钮是否可点击；
- 难以保证提交后彻底重置。

---

# 6. 各事件流程

## 6.1 补杠

流程：

```text
选择玩家
→ 点击补杠
→ 立即提交
```

点击次数：

```text
2 次
```

提交内容：

```ts
{
  type: "bugang",
  actorId
}
```

提交后：

- 清空状态；
- 更新分数；
- 显示反馈；
- 提供撤销。

---

## 6.2 暗杠

流程：

```text
选择玩家
→ 点击暗杠
→ 立即提交
```

点击次数：

```text
2 次
```

提交内容：

```ts
{
  type: "angang",
  actorId
}
```

---

## 6.3 直杠

如果计分规则需要记录放杠者，则流程为：

```text
选择杠牌者
→ 点击直杠
→ 在原玩家区选择放杠者
→ 立即提交
```

点击次数：

```text
3 次
```

进入第二位玩家选择状态后：

- 玩家位置不变；
- 已选杠牌者禁用；
- 其他玩家可选；
- 顶部提示改为“请选择放杠者”；
- 不弹出 Bottom Sheet；
- 不重排页面。

提交内容：

```ts
{
  type: "zhigang",
  actorId,
  counterpartyId
}
```

如果当前业务规则不需要记录放杠者，则可简化为两次点击。

Codex 在实现前应检查现有计分逻辑，不要擅自改变业务规则。

---

## 6.4 自摸

流程：

```text
选择玩家
→ 点击自摸
→ 启用番数区
→ 选择番数
→ 立即提交
```

点击次数：

```text
3 次
```

进入番数选择状态后：

- 玩家区位置不变；
- 自摸按钮保持选中；
- 其他事件按钮禁用或弱化；
- 番数区启用；
- 顶部提示显示“raisin 自摸，请选择番数”。

提交内容：

```ts
{
  type: "zimo",
  actorId,
  fan
}
```

不允许：

```text
选择玩家
→ 先选番数
→ 再点自摸
```

---

## 6.5 点炮

流程：

```text
选择胡牌者
→ 点击点炮
→ 在原玩家区选择点炮者
→ 启用番数区
→ 选择番数
→ 立即提交
```

点击次数：

```text
4 次
```

状态变化：

### 第一步：选择胡牌者

提示：

```text
已选择 raisin，请选择事件
```

### 第二步：点击点炮

提示：

```text
raisin 已胡牌，请选择点炮者
```

此时：

- raisin 禁用；
- 其他玩家可选；
- 玩家按钮位置保持不变；
- 番数区仍然禁用。

### 第三步：选择点炮者

提示：

```text
raisin 胡牌，张三点炮，请选择番数
```

此时番数区启用。

### 第四步：选择番数

立即提交：

```ts
{
  type: "dianpao",
  winnerId: actorId,
  loserId: counterpartyId,
  fan
}
```

---

## 6.6 流局

流局属于低频、复杂、批量结算操作。

不要强行套用普通事件的两到四次点击模型。

点击“流局”后进入专门的流局模式，但仍尽量复用当前固定页面区域。

可能涉及：

- 查花猪；
- 查大叫；
- 退刮风下雨；
- 多玩家选择；
- 批量结算；
- 结果预览。

建议结构：

```text
流局结算

查花猪
[玩家1] [玩家2] [玩家3] [玩家4]

查大叫
[玩家1] [玩家2] [玩家3] [玩家4]

退刮风下雨
[自动计算结果]

[确认流局结算]
```

流局可以保留一次最终确认，因为：

- 频率低；
- 影响多名玩家；
- 结算复杂；
- 错误成本高。

注意：

- “查花猪”“查大叫”“退刮风下雨”不应长期作为主页面常驻按钮；
- 它们只在进入流局模式后显示；
- 普通记分状态下主页面只显示六个主要事件。

---

# 7. 页面交互规则

## 7.1 初始状态

```text
提示：请选择玩家
玩家区：可用
事件区：禁用或可点击但提示先选玩家
番数区：禁用
反馈区：显示最近记录或为空
```

建议事件区在未选玩家时保持可见，但不可提交。

---

## 7.2 已选玩家

```text
提示：已选择 raisin，请选择事件
玩家区：当前玩家高亮
事件区：启用
番数区：禁用
```

---

## 7.3 等待番数

```text
提示：raisin 自摸，请选择番数
玩家区：保持原位置
事件区：自摸高亮，其他事件禁用
番数区：启用
```

---

## 7.4 等待第二位玩家

```text
提示：raisin 已胡牌，请选择点炮者
玩家区：原玩家禁用，其他玩家可选
事件区：点炮高亮
番数区：禁用
```

---

## 7.5 提交成功

```text
清除所有临时选择
恢复 idle
更新比分
显示“已记录”
允许撤销
```

---

# 8. 取消和返回机制

用户在中间状态必须可以取消当前录入。

推荐方式：

- 再次点击当前高亮玩家，取消整个录入；
- 点击顶部“取消”文字按钮；
- 按系统返回键时，先退出当前录入状态，不直接离开页面；
- 点击其他事件时，应重新开始当前事件流程，而不是叠加状态。

例如：

```text
raisin
→ 自摸
→ 当前等待番数
```

此时用户点击“点炮”：

建议行为：

```text
将当前事件切换为点炮
保留 actorId = raisin
进入选择点炮者状态
清空 fan
```

不要要求先手动取消再重新选择玩家。

---

# 9. 禁止出现的交互

Codex 实现时应避免以下行为：

## 9.1 禁止提前启用番数

错误：

```text
选择玩家后，番数立即可选
```

正确：

```text
只有点击自摸或完成点炮双方选择后，番数才启用
```

---

## 9.2 禁止频繁移动控件

错误：

- 点击事件后玩家区移动；
- 番数区插入页面导致事件区下移；
- Bottom Sheet 高度反复变化；
- 按钮位置随着状态重新排序。

正确：

- 玩家区、事件区、番数区固定占位；
- 只改变 enabled、disabled、selected 状态。

---

## 9.3 禁止所有事件都显示确认按钮

简单事件：

```text
玩家 → 补杠
```

点击补杠即提交。

不要再弹：

```text
是否确认补杠？
```

---

## 9.4 禁止提交后保留选中玩家

提交后必须完全清空。

---

## 9.5 禁止完全删除玩家昵称

动物头像是视觉识别增强，不是身份信息替代。

---

# 10. 视觉要求

## 10.1 点击目标

移动端按钮最小高度建议：

```text
44px
```

高频事件按钮建议：

```text
48px ~ 56px
```

玩家头像点击区域应包含：

- 头像；
- 昵称；
- 分数；
- 卡片留白。

不能只让头像圆形区域可点击。

---

## 10.2 状态颜色

至少区分：

- 默认；
- 已选择；
- 禁用；
- 危险或责任方；
- 成功反馈。

不要只靠红绿表达状态，应同时使用：

- 边框；
- 背景；
- 图标；
- 字重；
- 文案。

例如点炮者可以使用弱红色语义，但不能仅靠红色表示。

---

## 10.3 布局稳定

玩家区高度固定。

事件区高度固定。

番数区高度固定。

即使番数不可用，也保留该区域。

流局模式可以替换参数区内容，但不要改变玩家区与事件区的位置。

---

# 11. 推荐组件拆分

```text
GameEntryPanel
├── PlayerGrid
│   └── PlayerOption
├── EventGrid
│   └── EventButton
├── ParameterPanel
│   ├── FanSelector
│   └── DrawSettlementPanel
├── EntryStatusText
└── LastActionFeedback
```

建议逻辑层：

```text
useGameEntryMachine
```

负责：

- 当前状态；
- 玩家选择；
- 事件选择；
- 参数选择；
- 是否允许提交；
- 状态跳转；
- 提交；
- 重置；
- 取消；
- 撤销上一条。

不要把复杂状态逻辑分散在多个按钮组件内部。

# 12. 示例 TypeScript 接口

```ts
type EventType =
  | "zhigang"
  | "bugang"
  | "angang"
  | "dianpao"
  | "zimo"
  | "liuju";

type FanValue = 1 | 2 | 3 | 4;

type EntryDraft =
  | {
      eventType: "bugang" | "angang";
      actorId: string;
    }
  | {
      eventType: "zhigang";
      actorId: string;
      counterpartyId?: string;
    }
  | {
      eventType: "zimo";
      actorId: string;
      fan: FanValue;
    }
  | {
      eventType: "dianpao";
      actorId: string;
      counterpartyId: string;
      fan: FanValue;
    }
  | {
      eventType: "liuju";
      settlement: DrawSettlement;
    };
```

# 13. 验收标准

## 13.1 高频事件点击次数

| 事件 | 最大点击次数 |
|---|---:|
| 补杠 | 2 |
| 暗杠 | 2 |
| 直杠 | 2 或 3，取决于规则 |
| 自摸 | 3 |
| 点炮 | 4 |
| 流局 | 允许更多步骤 |

---

## 13.2 布局稳定性

录入过程中：

- 玩家按钮位置不变；
- 事件按钮位置不变；
- 番数按钮位置不变；
- 不出现连续 Bottom Sheet；
- 不因参数区启用而产生页面跳动。

---

## 13.3 状态合法性

以下状态不能出现：

```text
未选择事件，但番数已选择
补杠事件带有番数
点炮事件没有点炮者却允许提交
自摸事件没有番数却允许提交
同一玩家既是胡牌者又是点炮者
提交后仍有旧状态残留
```

---

## 13.4 反馈与撤销

每次成功提交后：

- 总分立即更新；
- 显示事件摘要；
- 提供撤销上一条；
- 录入界面立即恢复初始状态；
- 用户无需等待 Toast 消失即可继续操作。

---

# 14. 最终设计原则

Codex 实现时应始终遵循以下顺序：

```text
先优化高频路径
再处理低频例外
```

高频事件追求：

```text
少点击
短距离
固定位置
直接提交
```

低频复杂事件追求：

```text
语义明确
结果可预览
避免误操作
```

最终原则总结：

> 固定空间，渐进启用；  
> 事件决定参数，不允许参数提前介入；  
> 最后一个必要输入即提交；  
> 简单事件不确认，复杂事件可确认；  
> 提交后立即清空状态，并允许撤销。
