# 开发任务书（TASK）

> 本文档定义 mah-score v0.6.0 的开发任务。
>
> v0.6.0 的目标不是增加新的计分能力，而是优化线上真实环境中的房间创建、首屏加载和 Event 同步体验。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.6.0**

主题：

房间创建与 Event 同步体验优化。

目标：

在不改变 Event Sourcing、不引入 WebSocket、不更换基础设施的前提下，减少线上用户在以下场景中的等待和页面刷新感：

- 创建房间后进入房间
- 房间页首次加载
- 空 Event 同步
- 有新增 Event 的增量同步

本版本依据线上实测基线制定：

- 基线文档：`docs/iterations/experience-sync-latency/2026-07-13-baseline.md`
- 线上地址：`https://mah-score.vercel.app/`
- 测试房间：`338`

基线数据：

| 指标 | 当前实测 |
| --- | ---: |
| 首页入口 | 约 0.52s |
| 创建房间 | 约 3.10s |
| `/api/room` | 约 0.82s |
| `/api/room/events` | 约 0.95s |
| 空同步 `/api/room/sync` | 约 0.89s 到 0.93s |

要求：

- 保持 H5 优先
- 保持 Vercel Functions
- 保持 Vercel KV
- 保持 Event Sourcing
- Event Log 仍是唯一事实来源
- 不引入登录和用户系统
- 不引入新的后端平台
- 不引入复杂权限
- 不使用 WebSocket
- 不新增新的计分规则
- 不新增新的统计面板
- 不改变 Redis Key 模型
- 不复制业务逻辑到 UI

# 四、问题定义

## 房间创建慢

线上创建房间实测约 3.10 秒。

当前创建成功后，前端通过 `window.location.assign()` 进入房间页。这会重新加载整个前端应用，房间页再请求 room 和 events。

实际体验由以下部分叠加：

- 创建房间 API 等待
- 前端整页跳转
- 房间页应用重新初始化
- 房间首屏数据二次读取

## 房间首屏重复读取

当前房间页 `loadRoom()` 同时请求：

- `/api/room`
- `/api/room/events`

但后端 `getRoom()` 为了 replay 房间状态已经读取全量 events。

`/api/room/events` 为了校验房间存在，又调用 `getRoom()`，之后再读取 events。

因此首屏读取存在重复 Redis event list 读取。

## 空同步成本过高

线上空同步稳定约 0.89 到 0.93 秒。

当前 `/api/room/sync` 会：

1. 调用 `getRoom()`，读取全量 events 并 replay。
2. 调用 `readRoomEventsAfterVersion()`，再次读取全量 events。
3. 在内存中过滤出新增 events。

即使没有新增 event，也会产生全量 event log 读取和 replay 成本。

## 空同步导致页面刷新感

当前前端每 3 秒轮询一次，并设置：

- `syncStatus = "syncing"`
- `syncStatus = "idle"`

即使没有新增事件，也会触发 `RoomPage` 重新 render。

`RoomPage` render 内会计算：

- Replay
- Settlement
- Score History
- Player Ledger
- Round Ledgers

这会造成用户感知到页面刷新或状态闪动。

# Phase 1：房间首屏读取合并

## Epic 1

新增房间详情读取接口

### 目标

房间页首次加载时，只通过一次 API 请求拿到渲染所需的 room 和 events。

### 任务

新增或调整一个读取接口，返回：

- `room`
- `events`

建议优先新增：

`GET /api/room/detail?roomId=xxx`

原因：

- 不破坏现有 `/api/room` 和 `/api/room/events` 调用方。
- 便于小步上线和回滚。
- 符合 MVP 阶段保守改动原则。

后端实现要求：

- 不重复读取 event log。
- 一次读取 room hash。
- 一次读取 event list。
- 基于同一批 events replay 出 room 状态。
- 返回格式继续遵循统一 API 格式。

### 涉及模块

backend/api/room/detail.ts

api/room/detail.ts

backend/services/roomService.ts

shared/src/types/api.ts

frontend/src/api/roomApi.ts

frontend/src/pages/RoomPage.tsx

### API

新增：

GET /api/room/detail?roomId=xxx

响应：

```json
{
  "success": true,
  "data": {
    "room": {},
    "events": []
  },
  "message": ""
}
```

### Redis

不新增 Redis 结构。

不修改历史 Event。

不改变 Event Log 追加规则。

### 风险

room 状态必须仍然来自 replay，不能直接信任 Room Hash 中的 players。

旧接口仍然存在时，可能出现新旧读取路径并存。需要保证核心 replay 逻辑复用同一处实现。

### 验收标准

房间页首屏只调用一次房间详情接口。

接口返回 room 和 events。

room.players、room.status、room.version 与现有接口结果一致。

真实浏览器打开房间页可正常显示玩家、状态、历史和账本。

# Phase 2：sync 增量读取优化

## Epic 2

sync 接口避免全量 replay

### 目标

空同步不再读取全量 event log，不再 replay 全量房间状态。

### 任务

调整 `/api/room/sync`：

- 校验 `roomId`
- 校验 `version`
- 读取 room hash，确认房间存在
- 从 room hash 获取当前 `version`
- 当客户端 version 等于服务端 version 时，直接返回空 events
- 当客户端 version 小于服务端 version 时，只读取新增 events

新增 event 读取应尽量使用 Redis List index 范围：

- Event version 从 1 开始递增
- List index 从 0 开始
- `version=2` 后的新增事件，对应 list index 从 2 开始

即：

`LRANGE room:{roomId}:events version -1`

### 涉及模块

backend/api/room/sync.ts

backend/services/eventStore.ts

backend/services/roomService.ts

shared/src/types/api.ts

### API

保持：

GET /api/room/sync?roomId=xxx&version=xx

响应继续包含：

- `events`
- `version`

### Redis

不新增 Redis 结构。

继续使用：

- `room:{roomId}`
- `room:{roomId}:events`

### 风险

必须处理客户端 version 大于服务端 version 的异常情况。

必须保证版本号与 event list index 的映射正确。

历史脏数据或解析失败 event 不应导致接口崩溃。

### 验收标准

空同步返回：

- `events: []`
- `version: 当前服务端版本`

空同步不调用全量 replay。

有新增 event 时，只返回客户端 version 之后的 events。

连续 5 次空同步耗时记录到基线文档后续章节。

# Phase 3：前端增量应用 Event

## Epic 3

sync 返回新增 events 后直接追加

### 目标

当 sync 发现新增 events 时，前端直接把新增 events 追加到本地 events，而不是调用 `loadRoom()` 全量刷新。

### 任务

调整 `RoomPage` 同步逻辑：

- 使用当前本地最高 event version 发起 sync。
- sync 返回空 events 时，不更新 `events`。
- sync 返回新增 events 时，按 version 追加到本地 `events`。
- 更新 `roomVersion`。
- 必要时根据 replay 结果展示最新 room 状态。

需要避免：

- 重复追加同一个 event。
- events 顺序错乱。
- sync 请求并发返回导致旧响应覆盖新状态。

建议实现：

- 同一时间只允许一个 sync 请求在途。
- 或者对响应 version 做防旧响应保护。

### 涉及模块

frontend/src/pages/RoomPage.tsx

frontend/src/api/roomApi.ts

shared/src/domain/replay.ts

### API

不新增 API。

### Redis

不涉及。

### 风险

如果部分业务仍依赖 `room.status` 字段，而 status 只在 room state 中更新，可能出现 UI 状态滞后。

应优先让页面展示依赖 replayState，而不是依赖重复保存的 room 派生字段。

### 验收标准

其他玩家加入房间后，当前页面通过 sync 增量显示新玩家。

开始游戏、计分、撤销、确认本局、结束游戏都能通过增量 sync 更新页面。

有新增 events 时不触发 `/api/room/detail` 全量刷新。

空同步不改变 `events` 引用。

# Phase 4：减少空同步刷新感

## Epic 4

空同步不驱动整页 render

### 目标

没有新增 events 时，用户不应感知页面在刷新。

### 任务

调整同步状态展示策略：

- 空同步开始时不必立刻显示“同步中”。
- 只有用户主动刷新、首次加载、同步失败、或同步耗时异常时才更新明显 UI 状态。
- sync 成功且无新增 events 时，不更新会影响整个 RoomPage 的 state。

建议：

- 将 `syncStatus` 降级为非高频状态。
- 或拆到独立小组件，避免影响核心房间内容。
- 保留同步失败提示，不能吞异常。

### 涉及模块

frontend/src/pages/RoomPage.tsx

### API

不新增 API。

### Redis

不涉及。

### 风险

不能为了减少刷新感而隐藏真实同步失败。

用户主动刷新仍需要有明确反馈。

### 验收标准

无新增 events 的轮询不会造成页面文字闪动。

无新增 events 的轮询不会重算 replay、settlement、history、ledger。

断网或接口失败时仍能显示同步失败。

# Phase 6：创建房间体验优化

## Epic 6

创建成功后减少二次加载

### 目标

创建房间成功后，尽快进入可用房间页，减少重复请求和整页重载。

### 任务

优化方向按保守顺序执行：

1. `/api/room/create` 返回初始 room 和 events。
2. 前端创建成功后可复用初始 room/events。
3. 避免创建成功后立刻再做完整首屏读取。
4. 如需替换 `window.location.assign()`，先使用项目内轻量方案，不引入 React Router。

注意：

- 本 Epic 应在前面 sync 和 detail 接口稳定后再做。
- 不为了客户端路由引入新依赖。
- URL 最终仍需进入 `/room/{roomId}`，方便分享和刷新。

### 涉及模块

backend/api/room/create.ts

backend/services/roomService.ts

shared/src/types/api.ts

frontend/src/pages/HomePage.tsx

frontend/src/pages/RoomPage.tsx

frontend/src/App.tsx

### API

调整：

POST /api/room/create

响应在保留现有字段基础上，可增加：

- `room`
- `events`

### Redis

不新增 Redis 结构。

### 风险

需要保持现有创建房间调用兼容。

如果前端本地带入初始数据，需要确保刷新页面后仍能通过 detail 接口恢复。

### 验收标准

创建房间成功后进入房间可用状态的体感等待明显下降。

刷新 `/room/{roomId}` 后仍能正常读取房间。

邀请链接和手动加入不受影响。

# 八、退出条件

如果实现过程中发现以下情况，应暂停当前 Epic 并重新评估方案：

- 必须改变 Redis 数据模型才能继续。
- 必须引入 WebSocket 才能满足目标。
- 必须复制 shared/domain 业务逻辑到 UI。
- 必须引入新的状态管理库。
- 优化导致 Replay 与页面展示不一致。
- 优化导致撤销、结算或账本可信度下降。

---

# 九、最终原则

本次迭代的核心不是追求极限性能，而是减少明显浪费：

- 少一次请求
- 少一次 Redis 往返
- 少一次全量 replay
- 少一次无意义 render

所有改动都应服务于真实使用体验，并保持未来六个月后仍能一眼看懂。
