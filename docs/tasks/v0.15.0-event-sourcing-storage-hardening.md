# 开发任务书（TASK）

> 本文档定义 mah-score v0.15.0 的候选开发任务。
>
> v0.15.0 的目标是收口当前 Redis 存储与 Event Sourcing 基础模型，优先处理已确认的结构风险，不引入新的存储平台或复杂统计能力。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.15.0**

主题：

Event Sourcing 存储模型收口与 Redis 数据健康检查。

背景：

通过远程 Upstash Redis MCP 只读检查，当前数据现状为：

- `room:{roomId}` 使用 Hash；
- `room:{roomId}:events` 使用 List；
- Room Hash 当前主要字段为 `version`、`status`、`createdAt`、`updatedAt`；
- 未发现 Room Hash 中继续保存 `players`；
- 部分历史测试数据存在事件 payload 格式不一致，例如 `ROOM_CREATED` 的 `payload` 为数组；
- Upstash 当前未设置 TTL，历史测试房间长期保留。

当前阶段还没有真实用户使用，因此不优先处理长期统计、数据归档和图表能力。本任务只处理当前会影响 Event Sourcing 正确性、可维护性和排查效率的问题。

---

# 二、目标

本次目标：

- 让 RoomRecord 的业务状态尽量从 Event Log replay 得到；
- 降低 Room Hash 对业务状态的依赖；
- 兼容历史测试数据中的异常事件格式；
- 降低创建房间时产生半成品数据的风险；
- 增加只读 Redis 数据健康检查能力。

---

- 不删除历史房间；
- 不修改生产环境变量；
- 不引入新数据库；

- 不改三位房间号规则；

- 创建房间初始事件写入原子性；
- 只读健康检查脚本；
- 相关 domain/backend 测试或校验脚本。

# 四、当前急需处理的问题

## 1. 历史事件 payload 兼容

问题：

历史测试数据中存在 `ROOM_CREATED` 的 `payload` 为数组的情况。

当前风险：

- 当前事件解析器要求 `payload` 必须是普通 object；
- 异常事件会被丢弃；
- 房间可读性依赖旧的 Hash 快照兜底；
- Event Log 还没有完全成为唯一事实来源。

要求：

- 兼容历史 `ROOM_CREATED payload=[]`；
- 兼容逻辑必须局限在 parser 或明确的 legacy adapter 中；
- 不修改历史 Redis 数据；
- 新写入事件仍必须使用规范 payload。

验收标准：

- 含 `ROOM_CREATED payload=[]` 的旧房间仍能被 replay；
- 新事件写入不产生数组 payload；
- replay 校验脚本覆盖该兼容场景。

---

## 2. Room Hash 的 status 依赖

问题：

Room Hash 当前仍保存 `status`，且读取房间时仍要求 Hash 中存在合法 `status`。

当前风险：

- `status` 与 Event Log 存在重复；
- Hash 字段被当作业务事实的一部分；
- 未来删除或弱化 Hash `status` 时，读取逻辑会失败。

要求：

- `status` 优先由 Event Log replay 得到；
- Hash 中的 `status` 只能作为旧数据兼容或临时缓存；
- 不再让 `parseRoom()` 因缺少 Hash `status` 而直接判定房间不可读；
- 新模型下 Room Hash 应逐步退化为元数据和版本索引。

验收标准：

- 仅凭规范事件日志可以恢复 `WAITING`、`PLAYING`、`FINISHED`；
- 删除或缺失 Hash `status` 的测试样本仍可读取；
- 旧房间兼容逻辑有明确边界。

---

## 3. 创建房间初始写入原子性

问题：

当前创建房间流程是多步写入：

1. 创建 Room Hash；
2. 追加 `ROOM_CREATED`；
3. 追加首个 `PLAYER_JOINED`。

当前风险：

- 任一步失败都可能留下半成品房间；
- `version`、Hash、events List 可能短暂或永久不一致；
- 后续排查成本高。

要求：

- 创建房间时，Room Hash 与初始两条事件应由一次 Redis 原子操作完成；
- 初始版本应与事件列表一致；
- 仍需保证房间号未被占用；
- 失败时不留下可见半成品房间。

验收标准：

- 新建房间后 `room.version === LLEN(room:{roomId}:events)`；
- 初始事件顺序固定为 `ROOM_CREATED v1`、`PLAYER_JOINED v2`；
- 创建失败不会留下 `version=0` 或空事件列表房间。

---

## 4. Redis 数据健康检查脚本

问题：

当前 Redis 数据健康情况依赖人工 MCP 查询，不能稳定复现。

要求：

新增只读检查脚本，用于检查远程或本地 Redis 数据结构。

检查内容至少包括：

- `room:{roomId}` 是否为 Hash；
- `room:{roomId}:events` 是否为 List；
- Room Hash 字段是否在预期范围内；
- `room.version` 是否等于 events List 长度；
- events 是否都能解析；
- 第一条有效事件是否为 `ROOM_CREATED`；
- 是否至少存在一个 `PLAYER_JOINED`；
- 是否存在异常 payload 类型；
- 输出异常房间列表和简短原因。

要求：

- 脚本只读；
- 不删除、不修改 Redis；
- 不输出 Redis token；
- 可用于本地开发和远程 Upstash；
- 输出适合复制到 issue/task 复盘。

验收标准：

- 能在当前 Upstash 数据上运行；
- 能识别历史异常事件；
- 能输出总房间数、异常房间数和异常原因。

# 五、建议实施顺序

## 第一轮：健康检查脚本

目标：

先把当前数据问题可视化，避免继续靠人工抽样判断。

涉及：

- `backend/scripts/`
- `backend/services/redis.ts`
- `backend/services/eventStore.ts`

验收：

- 脚本能扫描 `room:*`；
- 脚本只读；
- 输出异常摘要。

## 第二轮：事件 parser 兼容

目标：

让旧事件格式不会破坏 replay。

涉及：

- `backend/services/eventStore.ts`
- `shared/src/domain/replay.ts`
- `shared/scripts/check-replay.mjs`

验收：

- `ROOM_CREATED payload=[]` 被兼容为 `{}`；
- 新事件仍保持规范 object payload。

## 第三轮：Room Hash status 退场第一步

目标：

让 `status` 不再是读取房间的必要业务事实。

涉及：

- `backend/services/roomService.ts`
- `shared/src/domain/replay.ts`
- 相关 API 返回类型。

验收：

- Room status 来自 replay；
- Hash status 缺失时，新结构房间仍可读；
- 旧房间仍兼容。

## 第四轮：创建房间原子化

目标：

消除新房间初始化的半成品风险。

涉及：

- `backend/services/roomService.ts`
- `backend/services/eventStore.ts`
- `backend/scripts/check-room-lock.mjs`

验收：

- 创建房间一次原子写入初始 Hash 和两条初始事件；
- 并发创建不会复用已有房间号；
- `version` 与 events List 长度一致。
