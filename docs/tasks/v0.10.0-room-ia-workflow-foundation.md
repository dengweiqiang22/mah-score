# 开发任务书（TASK）

> 本文档定义 mah-score v0.10.0 的开发任务。
>
> v0.10.0 的目标是根据 `docs/四川麻将计分器 - 信息架构（IA）重构方案.md`，先完成房间页工作流信息架构的基础落地。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.10.0**

主题：

房间页 IA 工作流基础重构。

目标：

让产品从“数据展示后台”进一步调整为“记录麻将事件的工具”，先完成页面职责拆分和主页面信息层级调整。

核心目标：

- 首页只回答“我要进入哪个房间？”
- 等待页只回答“人齐了吗？”
- 游戏页优先回答“当前比分如何、刚刚发生了什么？”
- 已结束页只回答“谁赢了？”
- 高频操作尽量在移动端首屏完成。
- 低频信息退出游戏主操作流。
- 保持现有计分逻辑、API、Event Sourcing、数据模型不变。

要求：

- Mobile First。
- Workflow First。
- 一个页面只负责一个核心任务。
- 不新增麻将规则。
- 不新增后端能力。
- 不调整 Redis 数据模型。
- 不复制 domain 业务逻辑到 UI。
- 不引入新的状态管理库。
- 不引入新的 UI 组件库。

# 二、开发原则

必须遵循：

小步调整。

先拆结构，再改交互。

每个 Phase 完成后页面必须可运行。

本版本优先级：

1. 页面任务清晰

3. 低频信息收纳

# 四、问题定义

## RoomPage 承担过多职责

当前 `RoomPage` 同时承担：

- 等待玩家
- 开始游戏
- 当前玩家识别
- 计分录入
- 本局账单
- 最近记录
- 历史局账单
- 玩家总账
- 邀请分享
- 房间管理
- 整场结算

这会导致游戏中页面默认信息量过大，用户需要先理解页面结构，才能完成一次计分记录。

## 游戏中页面不像高频工具

游戏中最重要的任务是：

刚刚发生了一件事，我要记录下来。

因此游戏中主页面不应默认展示完整账单、历史账单、总账和管理能力。它应该更像计算器或秒表，打开以后立即能完成高频动作。

## 本局账单是低频核对，不是高频入口

本局账单仍然重要，但它不是每次事件发生时都需要查看的信息。

主页面只保留最近两条记录用于确认和撤销，完整账单进入详情页面或次级入口。

# 五、目标信息架构

整体页面结构：

```text
首页
    │
    ▼
等待玩家
    │
    ▼
游戏中
    │
    ├── 本局详情
    └── 游戏结束
```

游戏中主页面：

```text
房间 754        第 1 局      菜单

张三        +12
李四         +3
raisin      -5
王五       -10

发生了什么？

自摸        点炮
杠牌        流局

最近记录
张三 自摸    +6        撤销
```

说明：

- 当前比分是一级信息。
- 事件录入是一级信息。
- 最近两条记录和撤销是二级信息。
- 本局账单、历史局、总账、房间管理是三级信息。

# Phase 1：首页入口信息层级调整

## Epic 1

调整首页为加入房间优先。

### 目标

让首页只回答“我要进入哪个房间？”

### 任务

- 保留房间号输入。
- 保留昵称输入。
- 加入房间为主按钮。
- 创建房间降级为次级入口。
- 邀请链接模式下隐藏房间号输入。
- 邀请链接模式下仍然只需要昵称即可加入。

### 涉及模块

`frontend/src/pages/HomePage.tsx`

`frontend/src/components/HomeActionButton.tsx`

### 验收标准

普通访问可以输入房间号和昵称加入。

邀请链接可以输入昵称加入。

创建房间仍可使用，但视觉权重低于加入房间。

错误提示仍清晰。

`npm run check` 通过。

---

# Phase 2：RoomPage 状态视图拆分

## Epic 2

拆分等待、游戏中、已结束视图。

### 目标

降低 `RoomPage` 的职责复杂度，为后续事件优先录入重构打基础。

### 任务

- `RoomPage` 保留数据加载、Replay、API handler 编排。
- 拆出 `WaitingRoomView`。
- 拆出 `PlayingRoomView`。
- 拆出 `FinishedRoomView`。
- 避免把业务计算复制到视图组件。
- 保持现有状态判断和 API 行为不变。

### 涉及模块

`frontend/src/pages/RoomPage.tsx`

`frontend/src/components/room/WaitingRoomView.tsx`

`frontend/src/components/room/PlayingRoomView.tsx`

`frontend/src/components/room/FinishedRoomView.tsx`

### 验收标准

等待、游戏中、已结束三个状态均可正常显示。

创建、加入、开始游戏、计分、撤销、确认下一局、结束整场行为不回归。

`RoomPage` 文件复杂度明显下降。

`npm run check` 通过。

---

# Phase 3：等待玩家页简化

## Epic 3

让等待页只服务“人齐了吗？”

### 目标

减少等待页设置感，保留开局前必要信息和操作。

### 任务

- 展示房间号。
- 展示 `x / 4 玩家`。
- 展示玩家昵称列表。
- 保留分享邀请。
- 保留开始游戏。
- 至少 2 人才能开始游戏的提示继续保留。
- 改名和删除玩家降级为低频入口或折叠区域。

### 涉及模块

`frontend/src/components/room/WaitingRoomView.tsx`

`frontend/src/pages/RoomPage.tsx`

### 验收标准

等待页首屏能看清房间号、人数、玩家列表和开始游戏入口。

分享邀请可用。

2 至 4 人可开始游戏。

玩家管理不再占据页面主视觉。

`npm run check` 通过。

---

# Phase 4：游戏中主页面 IA 重组

## Epic 4

让游戏主页面聚焦当前比分和事件录入。

### 目标

游戏中页面尽量一屏完成高频操作。

### 任务

- 顶部展示房间号、第几局和低频菜单入口。
- 当前比分只展示玩家昵称和总分。
- 事件录入区域成为页面核心。
- 最近记录最多展示两条。
- 撤销操作跟随最近记录。
- 移除主页面的本局账单完整展示。
- 移除主页面的历史局账单完整展示。
- 移除主页面的玩家总账完整展示。
- 手动刷新从主视觉移除。

### 涉及模块

`frontend/src/components/room/PlayingRoomView.tsx`

`frontend/src/components/room/RecordRow.tsx`

`frontend/src/components/room/PlayerTile.tsx`

`frontend/src/pages/RoomPage.tsx`

### 验收标准

移动端首屏能看到当前比分和事件录入入口。

游戏主页面默认不展示收入、支出、完整账单和历史账单。

最近记录最多两条。

最近记录仍可撤销。

本局已结束时仍可确认进入下一局。

`npm run check` 通过。

