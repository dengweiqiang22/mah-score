# 开发任务书（TASK）

> 本文档定义 mah-score v0.16.1 的候选开发任务。
>
> v0.16.1 的目标是优化流局结算录入体验，把流局状态录入并入主事件录入界面。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.16.1**

主题：

流局结算录入 UI 优化。

背景：

v0.16.0 已支持流局结算 payload 与 Replay 计分。

当前 UI 在点击“流局”后，会在高频记录区域下方额外列出一组流局结算列表。

问题：

- 页面高度增加；
- 玩家列表重复出现；
- 流局录入和普通事件录入的操作模型不一致；
- 用户需要在下方列表里核对每个玩家状态，扫视成本较高。

---

# 二、目标

本次目标：

- 将流局结算并入主事件录入界面；
- 不再额外渲染第二套流局玩家列表；
- 流局模式下采用“先选状态，再点玩家”的交互；
- 在玩家卡片右侧直接展示流局状态标签；
- 保持现有 `DRAW_GAME` payload 和 Replay 逻辑不变。

---

# 三、交互方案

## 普通事件模式

保持现有流程：

```text
玩家 -> 事件 -> 番数
```

适用于：

- 自摸；
- 点炮；
- 直杠；
- 补杠；
- 暗杠。

## 流局模式

点击“流局”后进入流局结算模式。

流局模式采用：

```text
流局状态 -> 玩家
```

状态段控：

```text
[花猪] [未叫] [有叫]
```

默认选中：

```text
花猪
```

操作规则：

- 花猪模式：点击玩家切换花猪；
- 未叫模式：点击玩家切换未叫；
- 有叫模式：先选最大番数，再点击玩家，标记该玩家听 N 番。

---

# 四、玩家状态标签

流局模式下，玩家卡片右侧必须展示当前流局状态。

优先级：

```text
已胡 > 花猪 > 未叫 > 听 N 番 > 未标记
```

展示：

- 已胡：`已胡`
- 花猪：图标 + 文案标签，例如 `🐷 花猪`
- 未叫：`未叫`
- 有叫：`听 2 番`
- 未标记：`未标记`

要求：

- 标签不是独立按钮；
- 整张玩家卡片仍是点击目标；
- 不仅依赖颜色表达状态；
- 花猪可以使用图标辅助，但必须保留文字；
- 已胡玩家置灰且不可点击。

---

# 五、互斥规则

状态互斥必须由 UI 自动处理：

- 标记花猪时，清除该玩家的未叫和听牌番数；
- 标记未叫时，清除该玩家的听牌番数；
- 标记听 N 番时，清除该玩家的未叫；
- 花猪玩家不能被标记未叫或听牌；
- 已胡玩家不能被标记任何流局状态。

---

# 六、确认区

流局模式下底部保留紧凑确认区：

```text
流局结算：花猪 1 · 未叫 1 · 有叫 2
[取消] [确认流局]
```

要求：

- 总数来自当前 UI 状态；
- 确认后继续提交现有 `DRAW_GAME` payload；
- 如果没有任何花猪、未叫、有叫标记，仍允许确认流局，表示无分数变化。

---

# 七、影响模块

预计涉及：

- `frontend/src/pages/RoomPage.tsx`
- `frontend/src/components/room/PlayerTile.tsx`
- 可能新增小型流局状态控件组件

不应涉及：

- `shared/src/domain/replay.ts`
- `shared/src/domain/history.ts`
- `backend/services/scoreValidation.ts`

除非发现现有 payload 生成存在 bug。

---

# 八、验收标准

- 点击流局后不再出现第二套玩家列表；
- 玩家卡片右侧能看到 `🐷 花猪`、`未叫`、`听 N 番`、`未标记`；
- 花猪、未叫、有叫的互斥逻辑正确；
- 有叫模式下，最大番数选择后可批量标记多个玩家；
- 已胡玩家不可操作；
- 确认流局提交的 payload 与 v0.16.0 一致；
- `npm run check` 通过；
- `npm run build` 通过。
