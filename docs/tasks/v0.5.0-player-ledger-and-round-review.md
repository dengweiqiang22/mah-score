# 开发任务书（TASK）

> 本文档定义 mah-score v0.5.0 的开发任务。
>
> v0.5.0 的目标不是增加新的计分规则，而是把房间页从“管理员总账视角”调整为“当前玩家视角 + 本局事件复盘”。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.5.0**

主题：

玩家视角账单与本局复盘优化。

目标：

让用户在真实牌局中可以快速完成三件事：

- 看到本局具体发生了哪些计分事件
- 清楚知道自己本局赚了多少或亏了多少
- 必要时查看全员账单和历史局复盘

核心产品原则：

- 事件流负责“发生了什么”
- 账单负责“算出来多少钱”
- 撤销挂在事件流上
- 默认视角是当前玩家
- 全员账单需要主动展开

要求：

- 保持 H5 优先
- 保持 Vercel Functions
- 保持 Vercel KV
- 保持 Event Sourcing
- 不引入登录和用户系统
- 不引入新的后端平台
- 不引入复杂权限
- 不使用 WebSocket
- 不新增新的计分规则
- 不新增新的统计面板
- 不改变 Redis 数据模型
- 不复制业务逻辑到 UI

# 四、概念定义

## 当前玩家

当前玩家是当前浏览器本地识别到的玩家。

MVP 阶段不做登录。

当前玩家身份来自：

- 创建房间返回的 `playerId`
- 加入房间返回的 `playerId`
- 本地存储保存的房间与玩家关系

如果当前浏览器无法识别玩家身份，则进入公共视图。

## 第几局

局表示四川麻将一局牌。

三家胡牌或流局后，本局结束。

用户确认账单后进入下一局。

## 第几笔

笔表示一局内的第 N 条有效计分动作。

例如：

- 第 1 局 · 第 1 笔：东风点炮胡
- 第 1 局 · 第 2 笔：西风自摸
- 第 1 局 · 第 3 笔：北风点炮胡

采用“笔”而不是“轮”，避免与麻将摸打轮次混淆。

## 本局事件

本局事件是当前局内发生过的计分动作。

包括：

- 点炮胡
- 自摸
- 直杠
- 补杠
- 暗杠
- 流局
- 已撤销的计分动作

撤销入口属于本局事件，不属于账单汇总。

## 我的账单

我的账单只展示当前玩家相关的收支。

默认展示我的账单，不默认展示全员账单。

## 全员账单

全员账单展示所有玩家在某一局中的收支。

必须由用户主动点击后展开。

# Phase 1：当前玩家身份

## Epic 1

当前玩家身份识别

### 目标

让房间页知道当前浏览器对应哪个玩家，从而默认展示“我的账单”。

### 任务

创建房间成功后：

- 保存 `roomId`
- 保存房主 `playerId`
- 保存房主昵称

加入房间成功后：

- 保存 `roomId`
- 保存加入者 `playerId`
- 保存加入者昵称

房间页加载时：

- 从本地存储读取当前房间对应的 `playerId`
- 校验该 `playerId` 是否仍存在于 Replay 后的玩家列表
- 如果存在，进入当前玩家视图
- 如果不存在，进入公共视图

本地存储只作为客户端身份辅助，不作为权限来源。

### 涉及模块

frontend/src/api/roomApi.ts

frontend/src/pages/HomePage.tsx

frontend/src/pages/RoomPage.tsx

frontend/src/utils

shared/src/types/api.ts

backend/api/room/create.ts

backend/api/room/join.ts

### API

POST /api/room/create

POST /api/room/join

### Redis

不新增 Redis 结构。

不把当前玩家身份写入 Redis。

### 风险

当前玩家身份不能被误用为权限系统。

本地存储可能丢失或过期，必须有公共视图兜底。

### 验收标准

创建房间后，当前浏览器能识别自己是房主。

加入房间后，当前浏览器能识别自己是加入者。

刷新房间页后，当前玩家身份仍然存在。

清空本地存储后，房间页进入公共视图，不报错。

当前玩家身份不影响后端接口校验。

# Phase 2：本局事件复盘

## Epic 2

本局事件流常驻展示

### 目标

让用户录入后立即看到本局具体发生了什么，能够复盘本局事件顺序。

### 任务

在快速录入区下方常驻展示“本局事件”。

每条事件卡片必须展示：

- 第几局
- 第几笔
- 操作类型
- 涉及玩家
- 分数变化
- 是否已撤销

示例：

`第 2 局 · 第 3 笔`

`西风直杠，东风引杠`

`西风 +1，东风 -1`

事件顺序：

- 最新事件在上方
- 或按发生顺序展示，但必须保持一致

建议默认最新事件在上方，方便刚录入后核对。

### 涉及模块

shared/src/domain/replay.ts

shared/src/domain/history.ts

shared/src/types/roomState.ts

frontend/src/pages/RoomPage.tsx

frontend/src/components

### API

无新增 API。

### Redis

不新增 Redis 结构。

### 风险

本局事件流不能重新发明一套与 Replay 不一致的计算逻辑。

“第几笔”必须来自事件顺序和撤销规则的稳定计算。

### 验收标准

录入点炮胡后，本局事件区出现对应事件。

录入自摸后，本局事件区出现第二笔事件。

录入杠牌后，本局事件区出现杠牌事件。

本局事件卡片显示第几局和第几笔。

本局事件展示与顶部分数一致。

## Epic 3

事件卡片支持撤销与已撤销标识

### 目标

让撤销功能回到正确位置：事件流。

用户撤销的是某一条事件，而不是某个玩家账单汇总。

### 任务

每条可撤销的有效事件卡片展示撤销按钮。

撤销确认文案必须包含：

- 第几局
- 第几笔
- 操作类型
- 涉及玩家

已撤销事件：

- 可以继续显示在事件流中
- 必须明确标识“已撤销”
- 不得显示可撤销按钮
- 不得影响任何账单汇总

示例：

`第 2 局 · 第 3 笔 · 已撤销`

`西风直杠，东风引杠`

`原变化：西风 +1，东风 -1`

### 涉及模块

shared/src/domain/history.ts

shared/src/domain/replay.ts

frontend/src/pages/RoomPage.tsx

api/room/undo.ts

backend/api/room/undo.ts

### API

POST /api/room/undo

### Redis

不新增 Redis 结构。

撤销仍通过追加 `UNDO` Event 实现。

### 风险

不能因为隐藏或标记已撤销事件，导致后续撤销目标错位。

撤销确认文案必须和实际请求的 `targetEventId` 一致。

### 验收标准

有效事件卡片可以撤销。

撤销确认文案包含具体事件信息。

撤销后事件卡片显示已撤销。

撤销后顶部总分回退。

撤销后我的账单和全员账单同步回退。

# Phase 3：当前玩家账单

## Epic 4

我的本局账单

### 目标

默认只展示当前玩家自己的本局账单，减少桌边计分时的信息干扰。

### 任务

当存在当前玩家身份时，本局账单默认展示：

- 我的本局收入
- 我的本局支出
- 我的本局净变化
- 与我相关的本局计分明细

不默认展示其他玩家账单。

当不存在当前玩家身份时，展示公共视图：

- 不强行假设当前玩家
- 可以展示本局事件
- 可以提供查看全员账单入口

我的本局账单必须位于本局事件下方。

### 涉及模块

shared/src/domain/history.ts

shared/src/domain/settlement.ts

frontend/src/pages/RoomPage.tsx

frontend/src/components

### API

无新增 API。

### Redis

不新增 Redis 结构。

### 风险

当前玩家身份丢失时不能导致账单空白或页面不可用。

### 验收标准

当前玩家进入房间后，默认只看到自己的本局账单。

我的本局账单与顶部总分变化一致。

与我无关的事件不出现在我的明细中，除非展开本局事件流。

无当前玩家身份时页面仍然可用。

# Phase 4：历史局复盘

## Epic 6

历史局账单与历史事件复盘

### 目标

让用户可以查看前面几局的账单和事件，而不是只能看玩家总账。

### 任务

当前账单下方展示历史局账单列表。

每个历史局卡片默认展示：

- 第几局
- 我的净变化
- 我的收入
- 我的支出

每个历史局可以展开：

- 该局事件流
- 我的该局账单明细
- 查看全员账单按钮

全员历史局账单同样必须按需展开。

历史局排序建议：

- 最近一局在上

### 涉及模块

shared/src/domain/history.ts

shared/src/domain/replay.ts

frontend/src/pages/RoomPage.tsx

frontend/src/components

### API

无新增 API。

### Redis

不新增 Redis 结构。

### 风险

历史局分组必须遵循 Replay 的局状态，不能按单条胡牌事件误判局数。

已撤销事件可以出现在事件流中，但不得影响账单。

### 验收标准

确认第 1 局账单后，进入第 2 局时能看到第 1 局历史账单。

历史局默认只展示我的账单摘要。

展开历史局后，可以看到该局事件流。

历史局全员账单默认隐藏，点击后显示。

历史局账单与最终结算一致。

# Phase 5：真实浏览器验收

## Epic 8

v0.5.0 真实浏览器验收流程

### 目标

使用真实浏览器验证当前玩家视角、本局事件流、撤销、我的账单、全员账单和历史局复盘。

### 任务

使用 Chrome DevTools MCP 或等效真实浏览器完成：

1. 打开首页
2. 创建房间，确认当前浏览器识别为房主
3. 通过邀请链接加入 3 名玩家
4. 开始游戏
5. 第 1 局录入点炮胡、自摸、第三家胡牌
6. 检查本局事件流显示 3 笔事件
7. 检查我的本局账单默认只显示当前玩家账单
8. 点击查看全员本局账单
9. 确认第 1 局账单
10. 进入第 2 局
11. 检查历史局列表中出现第 1 局
12. 展开第 1 局，检查该局事件流
13. 第 2 局录入暗杠
14. 第 2 局录入直杠
15. 从本局事件流撤销直杠
16. 检查直杠事件显示已撤销
17. 检查顶部总分、我的账单、全员账单一致
18. 结束游戏并查看最终结算

### 涉及模块

frontend/src/pages/HomePage.tsx

frontend/src/pages/RoomPage.tsx

frontend/src/components

shared/src/domain/history.ts

shared/src/domain/replay.ts

shared/src/domain/settlement.ts

backend/api/room/*

### API

覆盖现有房间 API。

不新增 API。

### Redis

不新增 Redis 结构。

### 风险

本版本主要改变页面信息结构，必须通过真实浏览器确认默认展开内容不会过载。

### 验收标准

真实浏览器完成完整流程。

无控制台业务错误。

当前玩家身份识别正确。

本局事件流可见且可撤销。

我的账单默认展示。

全员账单按需展开。

历史局账单可复盘。

最终结算与历史局账单一致。

# 七、建议执行顺序

建议执行顺序：

1. Epic 1：当前玩家身份识别
2. Epic 2：本局事件流常驻展示
3. Epic 3：事件卡片支持撤销与已撤销标识
4. Epic 4：我的本局账单
5. Epic 5：全员本局账单按需展开
6. Epic 6：历史局账单与历史事件复盘
7. Epic 7：玩家总账降级为折叠视图
8. Epic 8：真实浏览器验收流程

前三个 Epic 是本版本的基础。

没有当前玩家身份，就无法默认展示“我的账单”。

没有本局事件流，撤销和复盘都会缺少正确入口。

账单展示必须建立在事件流和 Replay 一致的基础上。
