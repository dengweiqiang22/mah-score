# 开发任务书（TASK）

> 本文档定义 mah-score v0.2.0 的开发任务。
>
> v0.2.0 的目标不是扩大系统复杂度，而是提升真实牌局中的使用体验。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.2.0**

主题：

真实牌局体验增强。

目标：

让用户在一场真实四川麻将中更快录入、更清楚复盘、更容易分享房间。

要求：

- 保持 H5 优先
- 保持 Vercel Functions
- 保持 Vercel KV
- 保持 Event Sourcing
- 不引入登录和用户系统
- 不引入新的后端平台
- 不引入复杂权限
- 不使用 WebSocket

---

# 二、开发原则

必须遵循：

小步提交

一次完成一个 Epic

每完成一个 Epic 必须能够运行

不得一次实现多个 Epic

复杂需求必须先分析方案，再开始开发。

本版本优先级：

1. 真实牌局可用性
2. 数据一致性
3. 交互效率
4. 可维护性
5. 功能完整度

# Phase 1：计分模型扩展

## Epic 1

四川麻将基础计分模型设计

### 目标

明确 v0.2.0 支持的计分事件、分值和 Replay 规则。

### 任务

定义：

- 点炮胡牌番数
- 自摸胡牌番数
- 点杠
- 明杠
- 暗杠
- 补杠
- 封顶

明确不做：

- 查叫
- 花猪
- 退税
- 呼叫转移
- 复杂牌型自动识别

### 涉及模块

shared/domain

shared/types

backend/services

frontend/pages

### 验收标准

有明确的数据模型。

有明确的分数计算规则。

不引入新数据库。

不引入新基础设施。

---

## Epic 2

扩展 Event Model

### 目标

为 v0.2.0 的计分项增加事件类型和 payload。

### 任务

新增或调整：

- 胡牌番数 payload
- 杠牌事件 payload
- 封顶字段

所有新增业务状态必须来自 Event Replay。

### 涉及模块

shared/types/event.ts

shared/types/score.ts

backend/services/eventValidation.ts

backend/services/scoreValidation.ts

### 验收标准

TypeScript 类型完整。

不使用 any。

旧事件仍可 Replay。

---

## Epic 3

Replay Engine 支持扩展计分

### 目标

Replay 能自动计算番数、杠分和封顶后的总分。

### 任务

支持：

- 点炮胡牌按番数计分
- 自摸胡牌按番数计分
- 点杠计分
- 明杠计分
- 暗杠计分
- 补杠计分
- 封顶

### 涉及模块

shared/src/domain/replay.ts

shared/scripts/check-replay.mjs

### 验收标准

Replay 测试覆盖新增计分项。

Undo 后分数正确。

流局后局数正确。

---

## Epic 4

计分 API 支持扩展计分

### 目标

后端 API 能写入新增计分事件，并拒绝非法事件。

### 任务

扩展：

POST /api/room/score

校验：

- 房间必须 PLAYING
- 玩家必须存在
- 胡牌玩家不能重复胡同一局
- 点炮人不能等于赢家
- 杠牌相关玩家必须合法
- 番数必须在允许范围内

### 涉及模块

backend/api/room/score.ts

backend/services/scoreValidation.ts

backend/services/eventStore.ts

### 验收标准

非法请求返回明确错误。

所有事件只追加，不修改历史事件。

# Phase 2：快速录入体验

## Epic 5

快速录入交互重做

### 目标

减少页面切换，让用户在房间页内快速完成计分。

### 任务

设计房间页内操作区：

- 操作类型选择
- 玩家选择
- 番数选择
- 杠牌类型选择
- 底部确认区
- 一键撤销上一条

交互目标：

- 点炮胡牌 3 到 4 次点击完成
- 自摸胡牌 2 到 3 次点击完成
- 点杠 3 次点击完成
- 流局 1 到 2 次点击完成

### 涉及模块

frontend/src/pages/RoomPage.tsx

frontend/src/api/roomApi.ts

### 验收标准

手机端可单手操作。

按钮不挤压、不换行异常。

不跳转页面即可完成计分。

# Phase 3：历史和撤销

## Epic 8

历史记录和 Undo 体验增强

### 目标

让用户能看懂每一条计分记录，并降低误操作成本。

### 任务

历史显示：

- 操作类型
- 相关玩家
- 番数
- 分数变化
- 是否已撤销
- 所属局数

Undo 支持：

- 撤销上一条
- 撤销指定计分事件
- 撤销前确认
- 历史中标记已撤销

### 涉及模块

backend/api/room/undo.ts

backend/services/eventStore.ts

frontend/src/pages/RoomPage.tsx

shared/src/domain/replay.ts

### 验收标准

Undo 仍然只追加 Undo Event。

不得删除历史 Event。

用户能通过事件列表复盘一局牌。

已撤销事件不从历史中消失。

手机端主要操作不需要频繁滚动。

# 五、需要补充确认的问题

以下问题需要在开发前确认：

1. 番数范围

默认建议：

1 番、2 番、3 番、4 番、封顶。

2. 封顶规则

需要确认：

按 4 番封顶，还是自定义最大倍数。

3. 杠牌计分

需要确认：

点杠、明杠、暗杠、补杠各自分值。

4. 结算文本格式

需要确认：

是否面向微信群直接复制粘贴。

5. 旧事件兼容

需要确认：

v0.1.0 的 SELF_DRAW、DISCARD_WIN 没有 fan 时是否默认按 1 番处理。

6. 二维码实现方式

需要确认：

是否允许新增轻量二维码依赖。

# 六、Definition of Done

每完成一个 Epic，必须满足：

TypeScript 无错误

ESLint 无错误

Replay 测试通过

不影响已有功能

不破坏 Event Sourcing 原则

不引入未经确认的新依赖

输出本次修改总结

不得进入下一个 Epic，直到当前 Epic 完成。
