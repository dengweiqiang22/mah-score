# 开发任务书（TASK）

> 本文档定义 mah-score v0.16.0 的候选开发任务。
>
> v0.16.0 的目标是补齐血战到底流局结算，并为房间级桌规配置预留数据模型空间。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.16.0**

主题：

流局结算与房间桌规配置基础。

背景：

当前 `DRAW_GAME` 只表示本局流局结束，不产生分数变化。

这满足 MVP 的“快速记账”，但不完整覆盖四川麻将血战到底中的流局规则。

实际牌桌还会涉及：

- 查花猪；
- 查叫；
- 退刮风下雨；
- 自摸是否额外加一档。

这些规则不应塞进高频记分页面的每次操作里，否则会增加用户心智负担。

房间级规则应在创建房间或房间设置中配置。

---

# 二、目标

本次目标：

- 补齐流局后的人工结算流程；
- 保持 mah-score 是计分工具，不做手牌识别；
- 默认采用“自摸加番”桌规；
- 为未来房间设置中的桌规配置预留数据模型；
- 所有分数仍必须来自 Event Replay。

---

# 三、默认桌规

## 自摸加番

默认规则：

```text
自摸时，在用户选择的基础番数上 +1 番。
```

项目内统一使用“自摸加番”，不引入“根”作为本阶段概念。

domain 字段建议使用：

```text
selfDrawExtraFan = 1
```

原因：

- 自摸加番只影响自摸事件；
- 根属于胡牌牌型附加番，依赖手牌判断；
- 当前阶段不做自动算番，也不录入根数。

示例：

```text
基础分 1
平胡自摸，用户选择 1 番
系统按 2 番结算
每个未胡玩家支付 2 分
```

4 番封顶时：

```text
用户选择 4 番
自摸加番后仍按 4 番封顶
每个未胡玩家支付 8 分
```

# 四、未来房间规则配置预留

当前可以先使用默认规则，不暴露设置 UI。

但事件和 domain 设计应预留未来配置空间。

候选配置：

```ts
interface RoomRuleConfig {
  readonly baseScore: 1;
  readonly maxFan: 4;
  readonly selfDrawExtraFan: 1;
  readonly enableDrawGameSettlement: boolean;
  readonly enableFlowerPig: boolean;
  readonly enableReadyHandCheck: boolean;
  readonly enableKongTaxRefund: boolean;
}
```

约束：

- 规则配置必须属于房间事实；
- 不应只保存在前端 localStorage；
- 不应复制到多个客户端实现；
- 新增客户端必须复用同一套 domain 逻辑。

建议事件：

```text
ROOM_CREATED payload.ruleConfig
```

未来如果允许中途改规则，应新增事件：

```text
ROOM_RULE_UPDATED
```

不要直接修改历史 `ROOM_CREATED` 事件。

# 五、流局结算范围

## 第一阶段：流局结算录入

点击“流局”后，不直接提交 `DRAW_GAME`。

进入流局结算面板，由用户人工确认：

- 谁是花猪；
- 谁已听牌；
- 听牌玩家最大番数；
- 谁需要退税。

系统不判断手牌。

## 第二阶段：查花猪

规则：

```text
花猪赔给每个非花猪玩家 9 倍基础分。
```

要求：

- 花猪玩家可多选；
- 已胡玩家是否参与，需要按实际桌规确认后再定；
- 计算必须写入 Event payload，并由 Replay 重新计算。

## 第三阶段：查叫

规则：

```text
未叫玩家赔给听牌玩家最大可能番型对应分数。
```

要求：

- 花猪不参与查叫；
- 听牌玩家需要选择最大番数；
- 未叫玩家赔给每个听牌玩家；
- 分数使用房间规则中的 `baseScore`、`maxFan`。

## 第四阶段：退刮风下雨

规则：

流局时，如果玩家是：

- 花猪；
- 未叫；

则退回本局全部杠牌所得。

要求：

- Replay 当前局内的有效 `KONG` 事件；
- 只退该玩家本局已经获得的杠牌收入；
- 撤销事件必须正确影响退税；
- 退税明细需要在最近记录和本局详情中展示清楚。

退税是本任务中风险最高的部分，应单独实现和测试。

# 六、事件模型建议

优先复用 `DRAW_GAME`，扩展 payload：

```ts
interface DrawGamePayload {
  readonly flowerPigPlayerIds: readonly string[];
  readonly readyHands: readonly {
    readonly playerId: string;
    readonly maxFan: number;
  }[];
  readonly notReadyPlayerIds: readonly string[];
  readonly kongTaxRefundPlayerIds: readonly string[];
}
```

原因：

- 流局仍然是一个本局结束事件；
- 不引入额外事件类型可以降低 replay 复杂度；
- payload 能表达流局结算所需人工事实。

兼容要求：

- 旧 `DRAW_GAME payload={}` 继续表示“流局无分数变化”；
- 新 payload 存在时才执行查花猪、查叫、退税。

# 八、验收标准

- 旧流局事件仍可 replay；
- 新流局事件可计算查花猪；
- 新流局事件可计算查叫；
- 退税逻辑有独立 replay 测试；
- 自摸默认额外 +1 番；
- 4 番封顶仍生效；
- 最近记录能展示流局结算明细；
- 本局详情能展示每个玩家的流局分数变化；
- Undo 流局后，本局状态和分数恢复；
- 不引入新存储平台；
- 不引入自动识别手牌逻辑。
