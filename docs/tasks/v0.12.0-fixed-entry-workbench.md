# 开发任务书（TASK）

> 本文档定义 mah-score v0.12.0 的候选开发任务。
>
> v0.12.0 的目标是根据 `docs/四川麻将计分器：高频事件录入交互设计文档.md`，将游戏中录入模块从“逐步确认流程”重构为“固定工作台”。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.12.0**

主题：

高频事件固定工作台录入。

目标：

降低牌桌上的高频录入成本，让熟练用户通过固定布局、渐进启用和最后必要输入自动提交，快速完成常见事件记录。

核心目标：

- 玩家区、事件区、番数区、反馈区固定位置。
- 玩家按钮、事件按钮、番数按钮在录入过程中不移动。
- 常见事件减少点击次数。
- 直杠、补杠、暗杠重新作为独立事件常驻。
- 自摸、点炮仅在事件确定后启用番数。
- 简单事件不显示确认按钮。
- 提交成功后清空所有临时状态。
- 提交成功后显示最近记录反馈，并允许撤销上一条。
- 保持现有计分逻辑、API、Event Sourcing、数据模型不变。

要求：

- Mobile First。
- 高频路径优先。
- 固定布局。
- 渐进启用。
- 最后一个必要输入即提交。
- 不使用连续 Bottom Sheet。
- 不新增后端 API。
- 不新增 Event 类型。
- 不调整 Redis 数据模型。
- 不引入新的状态管理库。
- 不复制 domain 计分逻辑到 UI。

# 二、背景说明

v0.11.0 已完成事件优先录入：

```text
选择事件
→ 选择玩家
→ 选择参数
→ 确认
```

但新的高频交互文档认为该方式更像逐步向导，不适合熟练用户反复录入。

v0.12.0 需要将核心录入模型调整为：

```text
选择玩家
→ 选择事件
→ 必要参数
→ 自动提交
```

同时保留 v0.10.0 的 IA 成果：

- 游戏中主页面聚焦计分。
- 低频账单、历史、邀请、管理能力不回到主流程。
- 不改变后端与 domain 数据模型。

# 三、本次开发范围

## 必须完成

固定工作台结构：

- 玩家区固定。
- 事件区固定。
- 番数区固定。
- 反馈区固定。

事件区固定展示六个事件：

- 直杠
- 补杠
- 暗杠
- 点炮
- 自摸
- 流局

建议排列：

```text
直杠  补杠  暗杠
点炮  自摸  流局
```

状态机实现：

- 使用显式状态机或等价的 discriminated union。
- 避免松散的 `selectedPlayer + selectedEvent + selectedFan` 组合。
- 状态必须能表达当前下一步任务。
- 提交后必须完全恢复 idle。

快速反馈：

- 提交成功后显示事件摘要。
- 反馈区提供撤销上一条。
- 反馈不阻塞后续录入。
- 用户无需等待反馈消失即可继续录入。

## 不开发

新的麻将计分规则。

查花猪、查大叫、退刮风下雨的完整流局结算。

新的后端接口。

新的 Redis Key。

权限系统。

动画重设计。

桌面端复杂布局。

# 四、问题定义

## 逐步确认流程点击成本偏高

v0.11.0 的流程强调清晰：

```text
事件
→ 玩家
→ 参数
→ 确认
```

但熟练用户在牌桌上更需要：

- 少点击；
- 短距离；
- 固定位置；
- 不反复搜索下一步；
- 不为简单事件额外确认。

## 杠牌合并不适合高频熟练用户

将直杠、补杠、暗杠合并为“杠牌”减少了主页面按钮数量，但会增加一次选择。

v0.12.0 以高频效率优先，因此三类杠牌重新作为独立事件常驻。

## 松散状态容易产生非法组合

以下状态不允许出现：

- 未选择事件但番数已选择。
- 补杠带有番数。
- 点炮没有点炮者却允许提交。
- 自摸没有番数却允许提交。
- 同一玩家既是胡牌者又是点炮者。
- 提交后仍残留上一笔状态。

因此录入状态必须显式建模。

# 五、目标交互

## 初始状态

```text
提示：请选择玩家
玩家区：可用
事件区：可见但不可提交，或点击后提示先选玩家
番数区：固定可见但禁用
反馈区：显示最近记录或为空
```

## 已选玩家

```text
提示：已选择 raisin，请选择事件
玩家区：当前玩家强高亮
事件区：启用
番数区：禁用
```

## 自摸

```text
选择玩家
→ 点击自摸
→ 选择番数
→ 自动提交
```

最大点击次数：

```text
3 次
```

## 点炮

```text
选择胡牌者
→ 点击点炮
→ 选择点炮者
→ 选择番数
→ 自动提交
```

最大点击次数：

```text
4 次
```

## 补杠

```text
选择玩家
→ 点击补杠
→ 自动提交
```

最大点击次数：

```text
2 次
```

## 暗杠

```text
选择玩家
→ 点击暗杠
→ 自动提交
```

最大点击次数：

```text
2 次
```

## 直杠

当前业务规则需要记录引杠玩家，因此流程为：

```text
选择杠牌者
→ 点击直杠
→ 选择引杠玩家
→ 自动提交
```

最大点击次数：

```text
3 次
```

## 流局

v0.12.0 先保持简单流局：

```text
点击流局
→ 确认流局
```

说明：

- 流局是低频且影响整局的操作，可以保留确认。
- 不在本版本实现查花猪、查大叫、退刮风下雨。
- 后续如果补齐复杂流局结算，应单独拆 task。

# 六、推荐技术方案

## 组件拆分

建议新增：

```text
frontend/src/components/room/GameEntryPanel.tsx
frontend/src/components/room/PlayerGrid.tsx
frontend/src/components/room/EventGrid.tsx
frontend/src/components/room/FanSelector.tsx
frontend/src/components/room/LastActionFeedback.tsx
frontend/src/hooks/useGameEntryMachine.ts
```

说明：

- 组件数量可以根据实际复杂度收敛。
- 不为了抽象而抽象。
- 状态机逻辑优先集中在 hook 或纯函数中。
- `RoomPage` 不应继续承载所有录入状态跳转细节。

## 状态机建议

建议采用：

```ts
type EntryState =
  | { readonly type: "idle" }
  | { readonly type: "player_selected"; readonly actorId: string }
  | { readonly type: "waiting_for_fan"; readonly actorId: string; readonly eventType: "SELF_DRAW" }
  | {
      readonly type: "waiting_for_counterparty";
      readonly actorId: string;
      readonly eventType: "DISCARD_WIN" | "DISCARD_KONG";
    }
  | {
      readonly type: "waiting_for_fan_after_counterparty";
      readonly actorId: string;
      readonly counterpartyId: string;
      readonly eventType: "DISCARD_WIN";
    }
  | { readonly type: "draw_confirm" };
```

说明：

- 使用现有 domain/API 命名，避免再引入 `zimo`、`dianpao` 等额外映射层。
- 直杠对应现有 `KONG + DISCARD_KONG`。
- 补杠对应现有 `KONG + SUPPLEMENT_KONG`。
- 暗杠对应现有 `KONG + CONCEALED_KONG`。

## 请求构造

继续构造现有 `ScoreEventRequest`：

- `SELF_DRAW`
- `DISCARD_WIN`
- `KONG`
- `DRAW_GAME`

不得新增 API。

# Phase 1：固定工作台骨架

## Epic 1

建立固定录入布局。

### 目标

玩家区、事件区、番数区、反馈区固定占位，录入过程中不发生布局跳动。

### 任务

- 将当前 `PlayingRoomView` 内的录入模块抽为 `GameEntryPanel` 或等价组件。
- 玩家区固定展示最多四名玩家。
- 事件区固定展示六个事件。
- 番数区始终可见。
- 反馈区始终有固定位置。
- 移除 v0.11.0 的确认摘要卡片作为常规提交路径。

### 涉及模块

`frontend/src/pages/RoomPage.tsx`

`frontend/src/components/room/PlayingRoomView.tsx`

`frontend/src/components/room/GameEntryPanel.tsx`

### 验收标准

录入过程中玩家按钮位置不变。

事件按钮位置不变。

番数按钮位置不变。

主页面不出现 Bottom Sheet。

`npm run check` 通过。

# Phase 2：显式录入状态机

## Epic 2

用显式状态机替代松散选择状态。

### 目标

避免非法状态组合，保证按钮启用、提示文案和提交动作来自同一状态。

### 任务

- 新增 `useGameEntryMachine` 或等价纯函数。
- 定义 `EntryState`。
- 定义玩家选择、事件选择、番数选择、取消、重置动作。
- 统一生成当前提示文案。
- 统一生成按钮 enabled / selected / disabled 状态。
- 提交成功后恢复 `idle`。

### 涉及模块

`frontend/src/hooks/useGameEntryMachine.ts`

`frontend/src/pages/RoomPage.tsx`

### 验收标准

未选择事件时番数不能选。

补杠和暗杠不会带番数。

点炮不会缺少点炮者。

同一玩家不能同时作为胡牌者和点炮者。

提交后状态完全清空。

`npm run check` 通过。

# Phase 3：高频事件自动提交

## Epic 3

实现直杠、补杠、暗杠、自摸、点炮的最后输入自动提交。

### 目标

减少确认步骤，让高频事件在最少点击内完成。

### 任务

- 补杠：玩家 + 补杠后立即提交。
- 暗杠：玩家 + 暗杠后立即提交。
- 直杠：玩家 + 直杠 + 引杠玩家后立即提交。
- 自摸：玩家 + 自摸 + 番数后立即提交。
- 点炮：胡牌者 + 点炮 + 点炮者 + 番数后立即提交。
- 自动提交期间禁用相关按钮。
- 提交失败时保留明确错误提示。

### 涉及模块

`frontend/src/hooks/useGameEntryMachine.ts`

`frontend/src/pages/RoomPage.tsx`

`frontend/src/api/roomApi.ts`

### 验收标准

补杠最多 2 次点击。

暗杠最多 2 次点击。

直杠最多 3 次点击。

自摸最多 3 次点击。

点炮最多 4 次点击。

高频事件不出现确认按钮。

`npm run check` 通过。

---

# Phase 4：固定反馈区与撤销

## Epic 4

实现提交反馈和撤销上一条。

### 目标

提交后用户立即知道记录成功，并能快速撤销上一条。

### 任务

- 成功后显示事件摘要。
- 反馈短时间后自动消失。
- 反馈不阻塞下一次录入。
- 反馈区提供撤销上一条。
- 最近记录仍最多展示两条。
- 撤销后重新加载房间事件。

### 涉及模块

`frontend/src/components/room/LastActionFeedback.tsx`

`frontend/src/components/room/RecordRow.tsx`

`frontend/src/pages/RoomPage.tsx`

### 验收标准

提交成功后显示类似 `已记录：raisin 自摸 · 1番`。

可以撤销上一条。

用户可以在反馈未消失时继续录入下一条。

撤销失败有明确错误提示。

`npm run check` 通过。

# Phase 5：流局保守处理

## Epic 5

保留简单流局确认，不引入复杂结算。

### 目标

让低频复杂操作避免误触，同时不扩大本版本范围。

### 任务

- 流局按钮常驻事件区。
- 点击流局进入 `draw_confirm`。
- 显示简单确认。
- 确认后提交现有 `DRAW_GAME`。
- 取消后恢复 `idle`。

### 涉及模块

`frontend/src/hooks/useGameEntryMachine.ts`

`frontend/src/pages/RoomPage.tsx`

### 验收标准

流局不会误触立即提交。

确认后仍使用现有 `DRAW_GAME`。

不实现查花猪、查大叫、退刮风下雨。

`npm run check` 通过。
