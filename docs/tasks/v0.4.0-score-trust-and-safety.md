# 开发任务书（TASK）

> 本文档定义 mah-score v0.4.0 的开发任务。
>
> v0.4.0 的目标不是增加新的计分能力，而是修复真实牌局试用中暴露的可信度问题和高风险误操作。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.4.0**

主题：

计分可信度与误操作修复。

目标：

让用户在完成一场真实牌局后，相信顶部总分、历史账本、本局账单和最终结算是一致的，并减少结束房间、删除玩家、撤销计分等高风险操作的误触。

本版本依据一次真实页面试用结果制定：

- 通过 `http://10.40.11.39:5173/` 创建房间
- 四名玩家加入
- 录入点炮、自摸、暗杠、直杠
- 撤销直杠
- 确认本局账单
- 结束游戏并查看结算

要求：

- 保持 H5 优先
- 保持 Vercel Functions
- 保持 Vercel KV
- 保持 Event Sourcing
- 不引入登录和用户系统
- 不引入新的后端平台
- 不引入复杂权限
- 不使用 WebSocket
- 不新增新的计分规则
- 不新增新的统计面板
- 不改变 Redis 数据模型

# Phase 1：计分可信度修复

## Epic 1

历史账本与撤销事件一致

### 目标

修复已撤销计分事件仍然进入玩家历史账本的问题。

用户展开历史记录时，玩家账本中的总分、收入、支出必须和顶部玩家分数保持一致。

### 背景

真实试用中执行：

1. 西风直杠
2. 东风引杠
3. 撤销该直杠

撤销后顶部总分正确回退：

- 东风：0
- 西风：6

但历史账本仍显示：

- 东风：-1
- 西风：+7

这会直接破坏用户对计分结果的信任。

### 任务

历史账本计算必须以有效事件为准。

当事件被 `UNDO` 撤销后：

- 不得计入玩家账本总分
- 不得计入玩家收入
- 不得计入玩家支出
- 不得作为可撤销收入记录继续显示

是否显示已撤销记录由实现方案决定，但必须满足：

- 如果显示，必须明确标记为“已撤销”
- 如果显示，不得影响任何收入、支出和总分

### 涉及模块

frontend/src/pages/RoomPage.tsx

shared/src/domain/replay.ts

shared/src/domain/settlement.ts

shared/scripts/check-replay.mjs

### API

无新增 API。

### Redis

不新增 Redis 结构。

不修改历史 Event。

撤销仍然通过追加 `UNDO` Event 完成。

### 风险

历史记录当前由前端页面局部函数计算，容易与 Replay 规则分叉。

如果继续在 UI 层重复计算账本，后续还可能再次出现与 Replay 不一致的问题。

### 验收标准

录入直杠后撤销，顶部总分必须回退。

展开历史记录后，玩家账本总分必须与顶部总分一致。

已撤销事件不得再显示为可影响账本的有效收支。

撤销后再次新增计分，历史账本仍然正确。

Replay 验收脚本覆盖 Undo 后账本一致性。

## Epic 2

最终结算局数统计修复

### 目标

修复最终结算把同一局内多次胡牌错误统计为多局的问题。

四川麻将血战到底中，一局可以出现多名玩家胡牌，但仍然属于同一局。

### 背景

真实试用中：

- 第 1 局出现 3 家胡牌并确认账单
- 第 2 局只录入暗杠和直杠撤销，未流局，未 3 家胡牌
- 随后结束游戏

最终结算显示：

`总局数：3`

该结果不符合真实牌局直觉。

### 任务

明确结算局数规则：

- 一局内多次胡牌只计 1 局
- `ROUND_CONFIRMED` 表示上一局已确认完成
- `DRAW_GAME` 所在局计为 1 局
- 三家胡牌自动结束的局计为 1 局
- 仅有杠牌但未流局、未三家胡牌的当前局，不计入已完成局数

结算文本和结算卡片必须使用同一套局数计算规则。

### 涉及模块

shared/src/domain/settlement.ts

shared/src/domain/replay.ts

frontend/src/pages/RoomPage.tsx

shared/scripts/check-replay.mjs

### API

无新增 API。

### Redis

不新增 Redis 结构。

### 风险

`rounds` 当前可能按计分事件记录，而不是按真实局记录。

修复时必须避免影响已有分数计算。

### 验收标准

一局内三家胡牌后，结算总局数只增加 1。

流局后，结算总局数增加 1。

只录入杠牌但未结束的一局，不计入已完成局数。

结算页展示的总局数与复制文本中的总局数一致。

Replay 验收脚本覆盖多胡一局、流局、杠牌未结束三种场景。

# Phase 2：高风险操作防误触

## Epic 3

结束游戏确认流程

### 目标

避免用户误点“结束游戏”后直接终止房间。

### 任务

点击“结束游戏”后必须二次确认。

确认文案需要明确后果：

- 房间会进入已结束状态
- 结束后不能继续计分
- 会生成最终结算

如果当前局处于进行中且已经存在本局计分事件，需要额外提醒：

- 当前局尚未流局
- 当前局尚未达到三家胡牌
- 当前局未确认账单

### 涉及模块

frontend/src/pages/RoomPage.tsx

api/room/event.ts

backend/api/room/event.ts

shared/src/domain/replay.ts

### API

POST /api/room/event

### Redis

不新增 Redis 结构。

### 风险

只做前端确认不能替代后端状态校验。

后端仍必须拒绝非法状态下的结束事件。

### 验收标准

点击“结束游戏”不会立即结束房间。

用户取消确认后，房间继续保持游戏中。

用户确认后，房间进入已结束状态。

当前局未完成时，确认文案必须包含明确提醒。

---

## Epic 4

删除玩家确认流程

### 目标

避免等待房间阶段误删玩家。

### 任务

点击“删除”后必须二次确认。

确认文案必须包含玩家昵称：

`确认删除玩家「南风」？`

删除只允许在房间等待开始阶段进行。

### 涉及模块

frontend/src/pages/RoomPage.tsx

api/room/player/remove.ts

backend/api/room/player/remove.ts

### API

POST /api/room/player/remove

### Redis

不新增 Redis 结构。

### 风险

删除玩家是等待阶段的管理操作，不应引入房主体系或复杂权限。

### 验收标准

点击删除不会立即删除玩家。

取消确认后，玩家仍在房间。

确认后，玩家才被删除。

游戏开始后，删除入口不可用。

---

## Epic 5

撤销确认文案增强

### 目标

让用户在撤销前知道具体会撤销哪一条计分记录。

### 任务

撤销上一条时，确认文案不再只显示：

`确认撤销上一条计分记录？`

需要显示具体目标，例如：

`确认撤销：第 2 局 西风直杠，东风引杠？`

从历史记录撤销某条记录时，也必须显示该记录的局次、玩家和操作。

### 涉及模块

frontend/src/pages/RoomPage.tsx

shared/src/domain/replay.ts

### API

POST /api/room/undo

### Redis

不新增 Redis 结构。

### 风险

撤销目标描述必须来自 Event Replay 或同源计算，不能与实际撤销目标不一致。

### 验收标准

撤销上一条时，确认框包含具体事件描述。

撤销历史记录时，确认框包含具体事件描述。

确认后撤销目标正确。

取消后不追加 `UNDO` Event。

# Phase 3：入口与可访问性修复

## Epic 6

邀请加入页聚焦加入任务

### 目标

用户通过邀请链接进入时，只完成加入房间，不被创建房间入口干扰。

### 背景

真实试用中访问：

`/?roomId=430`

页面隐藏了房间号输入框，这是正确的。

但同屏仍显示“创建房间”，扫码用户容易分心。

### 任务

当 URL 携带合法 `roomId` 时：

- 页面进入邀请加入模式
- 明确显示目标房间号
- 只展示昵称输入和加入按钮
- 不展示创建房间卡片

当 URL 未携带合法 `roomId` 时：

- 保持普通首页能力
- 支持创建房间
- 支持手动加入房间

当 URL 携带非法 `roomId` 时：

- 给出明确错误提示
- 不进入隐藏房间号的邀请模式

### 涉及模块

frontend/src/pages/HomePage.tsx

frontend/src/api/roomApi.ts

frontend/src/components/HomeActionButton.tsx

### API

POST /api/room/join

POST /api/room/create

### Redis

不新增 Redis 结构。

### 风险

邀请模式和普通模式不要复制两套复杂页面。

### 验收标准

邀请链接进入时，只看到加入房间相关内容。

页面上能看出正在加入哪个房间。

普通首页创建房间与手动加入能力不受影响。

非法邀请链接有明确提示。

# Phase 4：真实浏览器验收

## Epic 8

v0.4.0 真实浏览器验收流程

### 目标

建立一条固定的人工验收路径，确保本版本修复的问题能被真实页面验证。

### 任务

使用 Chrome DevTools MCP 或等效真实浏览器完成：

1. 打开首页
2. 创建房间
3. 通过邀请链接加入 4 名玩家
4. 开始游戏
5. 第 1 局录入点炮胡、自摸、第三家胡牌
6. 确认本局账单
7. 第 2 局录入暗杠
8. 第 2 局录入直杠并撤销
9. 展开历史记录
10. 检查历史账本与顶部总分一致
11. 点击结束游戏并取消确认
12. 再次点击结束游戏并确认
13. 检查结算总局数和复制文本

### 涉及模块

frontend/src/pages/HomePage.tsx

frontend/src/pages/RoomPage.tsx

shared/src/domain/replay.ts

shared/src/domain/settlement.ts

backend/api/room/*

### API

覆盖现有房间 API。

不新增 API。

### Redis

不新增 Redis 结构。

### 风险

只靠单元测试无法发现完整用户流程中的展示不一致。

真实浏览器验收不能替代 Replay 脚本，但必须作为本版本验收补充。

### 验收标准

真实浏览器完成完整流程。

无控制台业务错误。

历史账本与顶部总分一致。

结算总局数符合规则。

结束游戏和删除玩家都有确认。

邀请加入页只聚焦加入任务。
