# 开发任务书（TASK）

> 本文档定义 mah-score v0.13.0 的候选开发任务。
>
> v0.13.0 的目标是将房间页从“组件化后的原型页面”重构为一个统一、克制、适合牌桌单手操作的 H5 麻将计分工具。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.13.0**

主题：

房间页 UI 产品化重构。

核心判断：

当前项目已经有 TailwindCSS、shadcn/ui 风格本地组件、Lucide React，但页面仍然存在“只是替换组件，没有重组信息架构”的问题。

本次任务不是：

- 简单美化；
- 替换更多 shadcn 组件；
- 修改 className；
- 给按钮增加图标；
- 增加卡片和折叠区。

本次任务是：

- 建立项目级视觉规范；
- 重组高频记录页信息架构；
- 抽象 mah-score 产品级组件；
- 落实现有录入状态机的 UI 状态；
- 通过移动端截图进行验收和二次迭代；
- 将高频记录页沉淀为后续页面的视觉基准。

---

# 二、目标

将 mah-score 的房间页重构为：

> 现代、轻量、克制、适合线下牌桌单手操作的四川麻将计分工具。

用户打开游戏中页面后，应能第一眼知道：

- 当前房间；
- 当前局数；
- 同步状态；
- 每位玩家整场总分；
- 当前下一步如何记一笔；
- 最近发生了什么。

# 四、产品视觉定位

mah-score 不是：

- 企业后台；
- 管理系统；
- 传统红金棋牌大厅；
- 赌场风格；
- Material Design 默认风格；
- 大量渐变、阴影和装饰的展示型页面。

mah-score 应该是：

- 轻量；
- 稳定；
- 高效；
- 清晰；
- 温和；
- 有轻微麻将氛围；
- 适合牌桌旁快速操作。

# 五、设计原则

## 高频优先

游戏中页面优先服务：

1. 看分；
2. 记一笔；
3. 撤销最近错误；
4. 确认本局。

邀请、历史局、玩家总账、结束整场属于低频能力，不应占据主流程。

## 固定工作台

继续遵循 v0.12.0 的固定工作台思路：

```text
玩家
→ 事件
→ 必要参数
→ 自动提交
```

玩家区、事件区、番数区、反馈区在录入过程中保持稳定位置。

禁止：

- 点击一步后整块 UI 替换；
- 玩家按钮重新排序；
- 事件按钮移动；
- 番数区隐藏导致页面跳动；
- 每个事件都弹确认框。

## 产品组件优先

不要只在页面中堆：

```text
Button
Section
Disclosure
Card
Accordion
```

应抽象 mah-score 自己的产品组件：

```text
RoomHeader
RoomScoreStrip
PlayerTile
EventAction
FanSelector
EntryStatus
RecentEventRow
RoomOverflowMenu
ScoreValue
```

组件应表达产品语义，而不是只表达 HTML 结构。

```text

```

```text
主容器：16px
操作按钮：12px 或 14px
小标签：9999px
输入框：12px
```

阴影：

```text
只使用轻微阴影。

禁止每个模块都有明显投影。
```

验收标准：

- 文档能直接指导后续实现；
- Token 简单、可维护；
- 不新增依赖；
- 不引入复杂主题系统。

# 七、高频记录页重构任务

## Phase 2：重组 PLAYING / round_active 页面结构

当前游戏中页面不应继续表现为：

```text
高频记录 Card
最近记录 Card
本局详情 Accordion
更多 Accordion
```

目标结构：

```text
顶部房间状态
玩家总分区
当前状态提示
固定事件工作台
最近记录
低频功能入口
```

顶部房间状态：

- 房间号；
- 当前局数；
- 同步状态；
- 低频更多入口。

玩家总分区：

- 最多 4 位玩家；
- 头像占位或轻量识别符；
- 昵称；
- 整场总分；
- 当前用户“我”标签。

当前状态提示：

- 请选择玩家；
- 已选择某玩家，请选择事件；
- 已选择胡牌者，请选择点炮玩家；
- 自摸，请选择番数；
- 流局确认。

固定事件工作台：

- 玩家选择；
- 事件选择；
- 番数选择；
- 反馈区。

最近记录：

- 最多显示最近两条；
- 最新记录提供轻量撤销入口；
- 空状态保持轻量，不使用大块灰色占位。

低频功能：

- 邀请加入；
- 历史局；
- 玩家总账；
- 结束整场；
- 分享和复制。

低频功能可以先保留折叠区实现，但视觉权重必须低于主流程。后续可评估右上角菜单或本地轻量菜单组件。

验收标准：

- 首屏能看清房间、局数、玩家总分、事件区；
- 普通记分路径清楚；
- 用户不需要在多个同重量 Card 中寻找主操作；
- 低频能力仍然可用。

# 八、产品级组件任务

## Phase 3：抽象房间页产品组件

建议新增或重构：

```text
frontend/src/components/room/ScoreValue.tsx
frontend/src/components/room/RoomHeader.tsx

frontend/src/components/room/FanSelector.tsx
frontend/src/components/room/EntryStatus.tsx
frontend/src/components/room/RecentEventRow.tsx
frontend/src/components/room/RoomOverflowMenu.tsx
```

现有组件：

```text

frontend/src/components/room/LedgerRow.tsx
frontend/src/components/room/RecordRow.tsx
frontend/src/components/room/SettlementPlayerRow.tsx
```

可根据新视觉规范调整，但不得改变业务行为。

## 组件职责

### ScoreValue

- 统一显示 `+4`、`-6`、`0`；
- 统一正负分颜色；
- 使用 `tabular-nums`；
- 不让分数样式散落在页面各处。

### RoomHeader

- 展示房间号；
- 展示当前局数；
- 展示同步状态；
- 承载低频更多入口；
- 不重复展示当前用户名，当前用户在玩家区用“我”标签表达。

### RoomScoreStrip

- 展示整场玩家总分；
- 昵称截断；
- 分数清晰；
- 3 人和 4 人布局都不能出现明显空洞；
- 不展示本局收入、支出、最高分等复杂信息。

### PlayerTile

必须支持：

- 默认状态；
- 主体玩家状态；
- 责任玩家状态；
- 当前用户标签；
- 禁用状态；
- 已胡牌状态；
- 长昵称截断。

主体玩家用于：

- 胡牌者；
- 自摸玩家；
- 杠牌玩家。

责任玩家用于：

- 点炮者；
- 引杠玩家。

### EventAction

固定支持六个事件：

- 直杠；
- 补杠；
- 暗杠；
- 点炮；
- 自摸；
- 流局。

必须支持：

- 默认；
- selected；
- disabled；
- active；
- focus-visible。

可以使用 Lucide 图标辅助识别，但必须保留文字。

### FanSelector

- 固定显示 1 番、2 番、3 番、4 番；
- 默认禁用；
- 仅在状态机明确需要番数时启用；
- 禁用不隐藏；
- 选中状态清晰。

### EntryStatus

- 展示当前下一步；
- 固定高度，避免页面跳动；
- 不使用大型 Alert。

### RecentEventRow

- 用于最近记录和历史明细；
- 最近记录视觉轻量；
- 最新一条允许快速撤销；
- 更早记录弱化操作。

验收标准：

- `RoomPage.tsx` JSX 明显变薄；
- 组件表达产品语义；
- 不复制 domain 逻辑；
- 不使用 `any`；
- 组件状态清晰完整。

# 十、本局结算页任务

- 快速读懂本局输赢；

# 十七、最终原则

本次重构的核心不是“用了多少组件”，而是：

> 先建立 mah-score 的设计系统，再用 TailwindCSS、shadcn/ui 风格本地组件和 Lucide React 实现产品组件与完整页面。

房间页不是展示所有数据的后台，而是牌桌旁的高频计分工作台。

主页面只服务：

- 看分；
- 记一笔；
- 确认本局。

其他能力全部降级到次级区域。
