# 开发任务书（TASK）

> 本文档定义 mah-score v0.11.0 的开发任务。
>
> v0.11.0 的目标是在 v0.10.0 的 IA 基础上，完成“事件优先”的计分录入流程。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.11.0**

主题：

事件优先计分录入流程。

目标：

将游戏中页面的录入方式从“先选玩家，再选事件”调整为“先选事件，再选玩家，再选参数，再确认”。

核心目标：

- 用户在 3 秒内完成一次常见事件记录。
- 游戏主页面只展示四个事件入口。
- 杠牌入口合并，不直接展示暗杠、补杠、直杠三个按钮。
- 录入流程符合真实思考顺序。
- 提交前给出清晰确认信息。
- 记录成功后立即回到可继续录入状态。
- 保持现有计分逻辑、API、Event Sourcing、数据模型不变。

要求：

- Mobile First。
- 事件优先。
- 分步录入简单直接。
- 不在 UI 中复制计分规则。
- 不新增后端接口。
- 不新增 Event 类型。
- 不引入 Redux 或全局状态库。
- 不引入重型弹窗或动画库。

---

# 二、开发原则

必须遵循：

小步开发。

每次只落地一个事件流程。

每个流程完成后必须可运行和可回归。

本版本优先级：

1. 录入顺序符合用户思考
2. 防止误录和非法选择
3. 提交流程清晰
4. 记录后快速回到下一次录入
5. 代码可读、可维护
6. 视觉细节

# 三、本次开发范围

## 必须完成

游戏主页面事件入口改为：

- 自摸
- 点炮
- 杠牌
- 流局

事件录入流程改为：

```text
选择事件
    ↓
选择玩家
    ↓
填写参数
    ↓
确认
```

自摸流程：

- 选择自摸。
- 选择自摸玩家。
- 选择番数。
- 确认记录。

点炮流程：

- 选择点炮。
- 选择胡牌玩家。
- 选择点炮玩家。
- 选择番数。
- 确认记录。

杠牌流程：

- 选择杠牌。
- 选择暗杠、补杠或直杠。
- 选择杠牌玩家。
- 直杠时选择引杠玩家。
- 确认记录。

流局流程：

- 选择流局。
- 确认记录。

记录反馈：

- 记录成功后显示轻量反馈。
- 清空本次录入状态。
- 回到事件入口。

## 不开发

新的计分算法。

新的番数体系。

新的麻将规则。

房主权限。

操作员权限。

复杂动画。

WebSocket。

后端 API 调整。

Redis 数据模型调整。

# 四、问题定义

## 当前流程不符合真实思考顺序

当前游戏中页面要求用户先选玩家，再选点炮、自摸、直杠、补杠、暗杠、流局。

但真实牌桌上用户的第一反应通常是：

刚刚发生了什么？

而不是：

我要先选择哪个玩家？

因此录入流程应从事件开始。

## 杠牌入口过细

暗杠、补杠、直杠直接铺在主页面，会增加首屏按钮数量。

主页面只需要一个“杠牌”入口，具体杠类型在下一步选择。

## 主页面不应承担参数选择

番数、杠类型、相关玩家属于录入过程中的参数，不应常驻主页面。

这些信息应该在分步面板中出现，用完即退。

# 六、技术方案

采用本地轻量组件：

```text
ScoreEventPanel
ScoreEventStep
ScoreEventPreview
```

或根据实际代码保持更少组件。

建议状态模型：

```text
selectedEventType
selectedKongType
winnerId
discarderId
kongPlayerId
fromPlayerId
fan
```

原则：

- 状态只存在于游戏页面本地。
- 不引入全局状态。
- 构造请求仍使用现有 `ScoreEventRequest`。
- 提交仍调用现有 `recordScoreEvent`。
- 撤销仍调用现有 `undoRoomEvent`。

如果实现确认前分数预览，应优先在 shared domain 中新增纯 TypeScript helper，避免在 UI 中复制计分规则。

# Phase 1：事件入口与录入状态模型

## Epic 1

建立事件优先的入口和本地状态。

### 目标

主页面只展示四个事件入口，为分步录入做准备。

### 任务

- 将主页面入口调整为自摸、点炮、杠牌、流局。
- 删除主页面直杠、补杠、暗杠三个常驻按钮。
- 建立本地录入状态。
- 建立重置录入状态函数。
- 保持原有 API 请求构造能力。

### 涉及模块

`frontend/src/components/room/PlayingRoomView.tsx`

`frontend/src/pages/RoomPage.tsx`

### 验收标准

主页面只显示四个事件入口。

点击事件后进入对应录入状态。

取消或提交后能回到事件入口。

`npm run check` 通过。

---

# Phase 2：自摸和流局流程

## Epic 2

落地最简单的两个事件流程。

### 目标

先验证事件优先流程的基础交互和提交链路。

### 任务

自摸：

- 选择自摸玩家。
- 选择番数。
- 确认提交 `SELF_DRAW`。

流局：

- 点击流局后展示确认。
- 确认提交 `DRAW_GAME`。

### 涉及模块

`frontend/src/components/room/PlayingRoomView.tsx`

`frontend/src/api/roomApi.ts`

### 验收标准

自摸事件可正确记录。

流局事件可正确记录。

本局结束逻辑不回归。

记录成功后返回主入口。

非法状态下按钮不可提交。

`npm run check` 通过。

---

# Phase 3：点炮流程

## Epic 3

落地点炮事件录入。

### 目标

支持“谁胡牌、谁点炮、几番”的完整录入。

### 任务

- 选择胡牌玩家。
- 选择点炮玩家。
- 禁止胡牌玩家和点炮玩家相同。
- 禁止已胡玩家作为操作对象。
- 选择番数。
- 确认提交 `DISCARD_WIN`。

### 涉及模块

`frontend/src/components/room/PlayingRoomView.tsx`

### 验收标准

点炮事件可正确记录。

不能选择同一玩家作为胡牌和点炮。

已胡牌玩家不能继续作为胡牌玩家。

记录成功后最近记录更新。

`npm run check` 通过。

---

# Phase 4：杠牌流程

## Epic 4

落地合并后的杠牌事件录入。

### 目标

主页面只有“杠牌”，下一步再选择具体杠类型。

### 任务

- 选择杠牌。
- 选择暗杠、补杠或直杠。
- 选择杠牌玩家。
- 直杠时选择引杠玩家。
- 禁止杠牌玩家和引杠玩家相同。
- 禁止已胡玩家作为操作对象。
- 确认提交 `KONG`。

### 涉及模块

`frontend/src/components/room/PlayingRoomView.tsx`

### 验收标准

暗杠可正确记录。

补杠可正确记录。

直杠可正确记录。

直杠必须选择引杠玩家。

记录成功后最近记录更新。

`npm run check` 通过。

---

# Phase 5：确认反馈与预览

## Epic 5

完善记录确认和成功反馈。

### 目标

降低误操作风险，让记录完成后用户能立即继续看牌。

### 任务

- 提交前展示本次记录摘要。
- 记录成功后显示轻量反馈。
- 反馈自动消失。
- 记录成功后清空录入状态。
- 最近记录保持最多两条。
- 撤销上一条仍清晰可用。

如实现分数预览：

- 新增 shared domain helper。
- 不在 UI 中复制计分规则。
- 预览只作为确认辅助，不改变提交数据。

### 涉及模块

`frontend/src/components/room/PlayingRoomView.tsx`

`shared/src/domain/replay.ts`

`shared/src/domain/history.ts`

### 验收标准

提交前能看懂将要记录什么。

记录成功有明确反馈。

反馈不阻塞下一次录入。

最近记录和撤销不回归。

`npm run check` 通过。
