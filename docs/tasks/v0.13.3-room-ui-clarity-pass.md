# 开发任务书（TASK）

> 本文档定义 mah-score v0.13.3 的候选开发任务。
>
> v0.13.3 的目标是在 v0.13.0 - v0.13.2 已完成房间页产品化、低频功能收纳、玩家头像接入之后，继续做一轮 UI 清晰度优化。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.13.3**

主题：

房间页 UI 清晰度优化。

背景：

当前页面已经具备：

- 玩家头像选择；
- 玩家总分条；
- 高频记录工作台；
- 最近记录；
- 本局详情；
- 更多区域；
- 本局结算；
- 最终结算。

但从实际截图观察，仍有以下问题：

- 等待房间页没有贯穿头像识别；
- 录入状态提示在上方和下方重复；
- 事件明细仍然像日志，信息密度高；
- 房间管理的改名/删除文字按钮偏重；
- 部分事件按钮图标与麻将语义不够贴合。
- 当前头像资源来自 SVG 外壳包位图，构建产物偏大且不够干净；
- 用户已新增纯 SVG 向量头像目录 `frontend/src/assets/avatars/mah-score-vector-avatars/`。

---

# 二、目标

本次目标是提升房间页的“一眼识别”能力：

- 等待页也能通过头像识别玩家；
- 录入页不重复展示同一条提示；
- 事件明细更像计分明细，而不是系统日志；
- 工具型操作图标化，减少文字噪音；
- 事件按钮图标更贴近麻将动作语义。
- 将头像资源切换为纯 SVG 向量头像。

---

# 三、范围约束

## 不改变

- API；
- Event Sourcing；
- Redis 数据模型；

- 录入状态机；
- 计分规则；
- 同步机制；

## 可以改变

- 房间管理操作按钮样式；
- 高频记录工作台反馈区展示逻辑；
- 事件按钮图标；
- 事件明细行组件；
- 本局详情和事件明细的视觉层级。
- 前端头像资源引用路径；
- 头像 registry 中的 `src` 映射。

# 四、设计原则

## 头像识别贯穿

头像不只出现在游戏中页面。

应贯穿：

- 等待房间；
- 玩家总分条；
- 高频记录玩家区；
- 结算和历史中可根据需要逐步接入。

本轮必须完成：

```text
等待房间玩家列表展示头像
房间管理玩家行展示头像
```

## 头像资源使用纯 SVG

当前 v0.13.2 使用的头像文件位于：

```text
frontend/src/assets/avatars/*.svg
```

这些文件实际是 SVG 外壳包位图，构建产物较大。

本轮应切换为用户新增的纯 SVG 向量头像：

```text
frontend/src/assets/avatars/mah-score-vector-avatars/
```

要求：

- 更新 `frontend/src/utils/avatars.ts` 中的资源 import；
- 保持既有 avatar id 不变，避免影响已加入房间的 `avatarId`；
- 新头像目录下的 `avatars.json`、`README.md` 可作为参考，不直接作为运行时依赖；
- 不提交或引用 zip 文件；
- 不提交或引用 Windows `Zone.Identifier` 元数据文件；
- 如果旧头像资源确认不再使用，可在实现阶段评估删除旧 SVG 外壳位图文件，但删除前必须确认没有其他引用。

映射要求：

```text
panda      -> panda.svg
tiger      -> tiger.svg
cat        -> cat.svg
frog       -> frog.svg
monkey     -> monkey.svg
hamster    -> hamster.svg
pig        -> pig.svg
rabbit     -> rabbit.svg
bear       -> brown-bear.svg
dog        -> dog.svg
raccoon    -> raccoon.svg
lion       -> lion.svg
fox        -> fox.svg
cow        -> cow.svg
penguin    -> penguin.svg
koala      -> koala.svg
```

## 提示不重复

录入中页面上方已经有 `EntryStatus`：

```text
当前步骤
王五 自摸，请选择番数。
```

下方反馈区不应再次显示：

```text
当前录入
王五 自摸，请选择番数。
```

要求：

- 上方状态提示保留；
- 普通录入中，下方不再显示重复的“当前录入”；
- 下方反馈区只用于空状态、成功反馈、流局确认等必要内容；
- 流局确认仍可保留确认按钮，因为它需要明确提交。

## 工具入口可以图标化

适合图标化：

- 改名；
- 删除；
- 复制；
- 分享；
- 撤销；
- 展开/收起；
- 更多。

不适合只用图标：

- 开始游戏；
- 加入房间；
- 创建房间；
- 确认本局，开始下一局；
- 结束整场；
- 确认结束。

原则：

> 工具入口可以图标化，流程动作和危险确认必须保留文字。

本轮必须完成：

```text
房间管理的“改名 / 删除”入口改为图标按钮
删除确认仍保留明确文字
```

## 事件明细降噪

当前事件明细更像日志：

```text
8. 王五 自摸
未胡玩家每付 · 1 番 · 每家 1
王五 +1, raisin -1
```

目标是更像计分明细：

```text
[8] 王五 自摸        +1
    raisin -1        1番 · 每家1
```

要求：

- 每条优先展示序号、主体、事件、主要分数变化；
- 规则说明降为小字；
- 分数靠右或更容易扫读；
- 玩家名加粗；
- 不使用过重左侧竖线；
- 撤销入口保持低权重；
- 明细展开后仍能看清每条事件。

## 事件图标更贴合语义

当前部分图标语义偏弱：

```text
自摸：Sparkles
暗杠：Layers3
流局：RotateCcw
```

建议替换：

```text
自摸：Hand
暗杠：EyeOff
流局：CircleSlash
```

保留：

```text
直杠：ArrowRightLeft
补杠：Plus
点炮：Crosshair
```

要求：

- 文字标签必须保留；
- 图标只辅助识别；
- 不为六个事件使用六种颜色；
- 图标风格保持 Lucide 线性风格一致。

# 五、推荐组件

## WaitingPlayerRow

职责：

- 等待页玩家列表；
- 展示头像、昵称、玩家身份；
- 当前用户可显示轻量“我”标签；
- 用于替代纯文字玩家行。

## RoomManagementRow

职责：

- 房间管理玩家行；
- 展示头像和昵称；
- 改名图标；
- 删除图标；
- 使用 `aria-label` 保持可访问性。

## RoundEventDetailRow

职责：

- 事件明细行；
- 接收事件序号、标题、细节、分数流；
- 视觉上突出“谁做了什么”和主要结果；
- 保留撤销能力。

## EntryFeedbackPanel

可选。

如果当前 `RoomPage` 内反馈区逻辑仍复杂，可抽象。

职责：

- 空状态；
- 成功反馈；
- 流局确认；
- 普通录入中不重复展示当前步骤。

# 八、不能接受的结果

以下情况视为任务未完成：

- 只改颜色，没有降低信息噪音；
- 等待页仍不显示头像；
- 改名/删除图标没有 `aria-label`；
- 危险确认只剩图标没有文字；
- 普通录入中仍重复显示同一条提示；
- 事件明细仍然像日志堆叠；
- 图标替换后没有截图验收；
- 为 UI 清晰度调整修改计分规则或 API。
- 继续使用旧的 SVG 外壳位图头像；
- 提交 zip 文件或 `Zone.Identifier` 元数据文件。

# 九、最终原则

v0.13.3 的核心是：

> 减少不必要文字，保留必要确认；让玩家、动作、结果都更容易一眼识别。

页面不是越少文字越好，而是：

- 高频信息要明确；
- 工具操作要轻；
- 危险确认要清楚；
- 明细记录要可扫读。
