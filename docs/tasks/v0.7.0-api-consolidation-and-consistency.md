# 开发任务书（TASK）

> 本文档定义 mah-score v0.7.0 的开发任务。
>
> v0.7.0 的目标不是增加新的计分能力，而是把 Vercel API 数量收敛到免费额度内，并继续补强 Event Sourcing、并发一致性和验收能力。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.7.0**

主题：

API 数量收敛与数据一致性补强。

目标：

在不更换 Vercel、不更换 Vercel KV、不改变 Event Sourcing 架构的前提下，解决当前 Vercel 免费额度下 Serverless API 数量过多的问题，并把高风险写入路径进一步收敛。

核心目标：

- 将房间相关 API 收敛为少量 Vercel Functions
- 保持前端页面调用简单
- 保持 Event Log 是唯一事实来源
- 提高事件写入原子性
- 统一房间写操作并发保护
- 补齐关键 domain 验收测试
- 小步拆分 RoomPage 中最高维护压力的逻辑

要求：

- 保持 H5 优先
- 保持 Vercel Functions
- 保持 Vercel KV
- 保持 Event Sourcing
- 保持 Polling，不使用 WebSocket
- 不引入新的后端平台
- 不引入登录、用户系统、复杂权限
- 不引入重型测试框架
- 不新增计分规则
- 不新增统计面板
- 不复制业务逻辑到 UI

# 四、问题定义

## Vercel API 数量过多

当前项目 root `api/` 下存在多个房间相关 Function wrapper，例如：

- `/api/room/create`
- `/api/room/join`
- `/api/room/start`
- `/api/room/score`
- `/api/room/undo`
- `/api/room/events`
- `/api/room/sync`
- `/api/room/player/rename`
- `/api/room/player/remove`
- `/api/room`
- `/api/health`

Vercel 免费额度对 Serverless API 数量存在限制。当前拆分方式不利于长期维护和部署。

## 事件写入不是单个原子操作

当前事件追加流程包含：

1. Room Hash `version +1`
2. Event List `RPUSH`
3. Room Hash `updatedAt/status` 更新

这些步骤应尽量收敛为单个原子写入流程，避免中间步骤失败导致 Room Hash 与 Event Log 不一致。

## 写接口并发保护不统一

部分写接口已经使用房间锁，部分写接口仍需要统一收口。

所有会追加 Event 的房间写操作，都应遵循同一套流程：

1. 获取房间锁
2. 锁内读取最新 Room 和 Events
3. 锁内 Replay
4. 锁内业务校验
5. 锁内追加 Event
6. 释放房间锁

## RoomPage 维护压力偏高

`RoomPage` 同时承担：

- 房间加载
- Event 同步
- 二维码生成
- 玩家编辑
- 快捷计分
- 撤销
- 历史账单
- 玩家流水
- 结算复制

本版本只拆最高收益逻辑，不进行大规模 UI 重构。

# 五、开发阶段

---

# Phase 1：Vercel API 数量收敛

## Epic 1

房间 API 统一入口设计与落地。

### 目标

将房间相关 API 收敛到单个 Vercel Function 入口，减少免费额度占用。

### 推荐方案

保留：

- `api/health.ts`
- `api/room.ts`

房间业务统一进入：

```text
GET /api/room?roomId=123
GET /api/room?action=events&roomId=123
GET /api/room?action=sync&roomId=123&version=10

POST /api/room
{
  "action": "create",
  ...
}

POST /api/room
{
  "action": "join",
  ...
}
```

POST action 包括：

- `create`
- `join`
- `start`
- `score`
- `undo`
- `event`
- `renamePlayer`
- `removePlayer`

### 涉及模块

`api/room.ts`

`backend/api/room/*`

`frontend/src/api/roomApi.ts`

`vercel.json`

### 验收标准

root `api/` 下 Function 数量控制在免费额度内。

前端所有房间操作均通过新入口调用。

旧 wrapper 不再作为部署入口存在。

`npm run check` 通过。

本地创建房间、加入房间、开始游戏、计分、撤销、同步均可用。

---

# Phase 2：事件写入原子性补强

## Epic 2

收敛 append event 写入流程。

### 目标

让事件追加尽量成为一个原子写入单元。

### 方案优先级

1. Redis transaction
2. Redis Lua script
3. 保留锁，但补充失败检测和错误提示

优先选择 Upstash Redis 当前 SDK 支持且代码最容易维护的方案。

### 涉及模块

`backend/services/eventStore.ts`

`backend/services/redis.ts`

`backend/services/roomService.ts`

### 验收标准

新增 Event 时不会出现重复版本号。

新增 Event 时不会出现 `version` 已增加但 Event 未写入的正常成功返回。

`updatedAt` 与追加 Event 的 timestamp 一致。

`GAME_STARTED`、`GAME_FINISHED` 状态缓存仍正确更新。

---

# Phase 3：写接口统一房间锁

## Epic 3

统一房间写操作并发保护。

### 目标

所有会追加 Event 的写接口走同一套房间锁和锁内校验流程。

### 涉及操作

- 创建房间
- 加入房间
- 修改昵称
- 删除玩家
- 开始游戏
- 计分
- 撤销
- 追加房间事件

### 涉及模块

`backend/services/roomService.ts`

`backend/services/eventStore.ts`

`backend/api/room/*`

### 验收标准

同时加入房间不会丢玩家。

同时开始游戏不会重复开始。

同时计分不会产生重复版本号。

同时撤销不会撤销同一个事件两次。

所有写接口对 `ROOM_BUSY` 返回统一可理解错误。

---

# Phase 4：Event Sourcing 补齐收尾

## Epic 4

确认房间生命周期完全可 Replay。

### 目标

让核心房间状态都能从 Event Log 恢复。

### 必须支持的事件

- `ROOM_CREATED`
- `PLAYER_JOINED`
- `PLAYER_RENAMED`
- `PLAYER_REMOVED`
- `GAME_STARTED`
- `GAME_FINISHED`
- `ROUND_CONFIRMED`
- `DISCARD_WIN`
- `SELF_DRAW`
- `KONG`
- `DRAW_GAME`
- `UNDO`

### 涉及模块

`shared/src/domain/replay.ts`

`shared/src/types/event.ts`

`backend/services/roomService.ts`

`frontend/src/pages/RoomPage.tsx`

### 验收标准

Replay 可恢复玩家列表、房间状态、当前局、历史局、分数。

Room Hash 只作为最小缓存使用。

旧房间兼容策略保留，并在代码或文档中说明边界。

前端不需要自行伪造新模型事件。

## Epic 5

### 目标

### 测试范围

- 点炮

- 多人血战到底当前局状态

### 涉及模块

`shared/src/domain/*`

`shared/scripts/check-replay.mjs`

`package.json`

### 验收标准

`npm run check:replay` 覆盖本版本新增场景。

`npm run check` 通过。

`npm run check:acceptance` 通过或明确说明无法运行原因。

# Phase 6：RoomPage 小步拆分

## Epic 6

拆分房间同步逻辑。

### 目标

降低 `RoomPage` 维护压力，但不改变 UI 和业务交互。

### 推荐范围

优先抽出：

- `useRoomSync`
- 必要的同步辅助函数

暂不拆：

- 计分 UI
- 历史 UI
- 结算 UI

除非这些拆分是为了完成同步逻辑抽离所必需。

### 涉及模块

`frontend/src/pages/RoomPage.tsx`

`frontend/src/hooks/useRoomSync.ts`

`frontend/src/api/roomApi.ts`

### 验收标准

页面表现不变。

同步行为不变。

`RoomPage` 中同步相关状态和 effect 明显减少。

`npm run check` 通过。

# 六、验收命令

本版本每个 Epic 至少执行：

```bash
npm run check
```

涉及 Replay 或计分逻辑时执行：

```bash
npm run check:replay
```

涉及并发或锁逻辑时执行：

```bash
npm run check:room-lock
```

最终验收执行：

```bash
npm run check:acceptance
```

# 七、完成标准

v0.7.0 完成时必须满足：

- Vercel Function 数量在免费额度内
- 房间相关 API 入口已收敛
- 前端调用新 API 入口
- 事件写入流程有原子性保障或明确的最小风险实现
- 写接口并发保护统一
- Event Replay 能恢复完整核心房间状态
- Domain 验收覆盖关键计分和撤销场景
- RoomPage 同步逻辑已小步拆分
- 不引入新基础设施
- 不引入复杂权限和用户系统
