# 开发任务书（TASK）

> 本文档定义 mah-score v0.3.1 的开发任务。
>
> v0.3.1 的目标不是增加新的计分能力，而是把现有房间操作流程修顺，减少误点、减少切换、提高可读性。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.3.1**

主题：

房间操作体验修复。

目标：

让用户在进入房间、识别玩家状态、录入直杠、查看历史时更直接、更少误操作。

要求：

- 保持 H5 优先
- 保持 Vercel Functions
- 保持 Vercel KV
- 保持 Event Sourcing
- 不引入登录和用户系统
- 不引入新的后端平台
- 不引入复杂权限
- 不使用 WebSocket
- 不新增新的计分能力
- 不新增新的统计面板

---

# 二、开发原则

必须遵循：

小步提交

一次完成一个 Epic

每完成一个 Epic 必须能够运行

不得一次实现多个 Epic

复杂需求必须先分析方案，再开始开发

本版本优先级：

1. 降低误操作
2. 降低页面切换
3. 提高状态可见性
4. 保持底座稳定
5. 保持实现简单

# Phase 1：邀请进入流程

## Epic 1

邀请房间进入入口简化

### 目标

用户通过扫码或复制链接进入房间时，只需要输入昵称即可，不再被房间号输入干扰。

### 任务

当 URL 已携带 `roomId` 时：

- 进入邀请加入模式
- 不显示房间号输入框
- 仅保留昵称输入和加入按钮
- 房间号作为隐藏上下文从 URL 获取

当 URL 未携带 `roomId` 时：

- 保持现有首页能力
- 继续支持创建房间与手动加入房间

### 涉及模块

frontend/src/pages/HomePage.tsx

frontend/src/api/roomApi.ts

frontend/src/components/HomeActionButton.tsx

### API

POST /api/room/create

POST /api/room/join

### Redis

不新增 Redis 结构。

### 风险

邀请模式与普通模式的页面状态可能分叉。

邀请链接参数非法时需要保持明确错误提示。

### 验收标准

扫码或复制链接进入首页后，不再看到房间号输入框。

用户可以直接输入昵称加入房间。

普通首页入口不受影响。

## Epic 2

玩家状态与已胡牌标识优化

### 目标

让用户在录入时一眼看出哪些玩家已经胡牌或已经退出当前局，避免点错。

### 任务

玩家卡片需要明确展示当前状态：

- 未胡牌
- 已胡牌

已胡牌玩家在视觉上需要与可操作玩家区分开。

在可录入场景中：

- 已胡牌玩家不能被当作有效目标继续录入
- 已胡牌玩家的状态要和当前回合状态一致

### 涉及模块

frontend/src/pages/RoomPage.tsx

shared/src/domain/replay.ts

shared/src/types/roomState.ts

### API

无新增 API。

### Redis

不新增 Redis 结构。

### 风险

已胡牌状态必须严格来自 Replay，不能由前端临时猜测。

撤销后玩家状态需要同步回退。

### 验收标准

页面上能明确识别已胡牌玩家。

已胡牌玩家在录入交互中不会被误选为有效目标。

撤销后状态显示能够恢复正确。

# Phase 2：录入交互

## Epic 3

直杠录入流程重构

### 目标

把直杠从“先选玩家，再往下找引杠人”的流程，改成更短、更适合手机单手操作的流程。

### 任务

直杠录入顺序调整为：

1. 选择直杠玩家
2. 选择操作“直杠”
3. 在同一操作区域继续选择引杠玩家

交互上需要做到：

- 主玩家用一种颜色标识
- 引杠玩家用另一种颜色标识
- 候选玩家保持在上方可见区域
- 不依赖页面下滑去完成核心步骤

### 涉及模块

frontend/src/pages/RoomPage.tsx

shared/src/domain/replay.ts

shared/src/types/score.ts

### API

POST /api/room/score

### Redis

不新增 Redis 结构。

### 风险

快速录入状态机容易被一次性改乱。

直杠流程调整后，其他计分模式不能被连带破坏。

### 验收标准

直杠可以在短路径内完成录入。

主玩家和引杠玩家有明确视觉区分。

不需要把手指大幅移到页面下方才能继续下一步。

# Phase 3：历史记录

## Epic 4

历史记录按玩家视角展示

### 目标

把历史记录从统一事件流，改成玩家更容易理解的收入 / 支出视图。

### 任务

历史区展示内容改为：

- 每个玩家的收入
- 每个玩家的支出
- 每条记录对应的局次和对局对象
- 必要时保留撤销入口，但不把原始事件日志作为主视图

展示上需要让用户更快回答这些问题：

- 我这一局赚了多少
- 我这一局亏了多少
- 是谁付给了谁

### 涉及模块

shared/src/domain/replay.ts

shared/src/domain

frontend/src/pages/RoomPage.tsx

shared/src/types

### API

无新增 API。

### Redis

不新增 Redis 结构。

### 风险

事件日志到玩家账本的映射关系必须准确。

Undo 仍然需要能定位到原始事件。

### 验收标准

历史记录主视图不再是原始事件列表。

用户可以从历史区直接看懂每个玩家的收入和支出。

撤销功能仍然可用。

# Phase 4：本局结算流程

## Epic 5

本局结束后的账单确认与下一局进入流程

### 目标

当一局结束后，先展示本局账单并等待用户确认，再进入下一局录入。

### 任务

当本局结束条件满足时：

- 三名玩家胡牌后结束本局
- 或流局后结束本局

需要展示：

- 本局结束原因
- 本局账单或结算摘要
- 当前局的玩家收支结果

在用户确认前：

- 不进入下一局录入
- 不继续录入当前局的计分事件
- 保持当前局结果可见

在用户确认后：

- 进入下一局
- 清空当前局录入状态
- 保留房间总状态和历史事件

### 涉及模块

frontend/src/pages/RoomPage.tsx

shared/src/domain/replay.ts

shared/src/types/roomState.ts

shared/src/domain/settlement.ts

### API

无新增 API。

### Redis

不新增 Redis 结构。

### 风险

本局状态与房间状态容易混淆，导致确认后错误推进。

如果结算展示与 Replay 不一致，用户会不信任账单。

下一局开始条件必须清晰，避免在结束局上继续录入。

### 验收标准

本局结束后会先展示账单或结算摘要。

用户确认后才进入下一局。

当前局结束后不能继续追加当前局录入。

房间历史事件和总分不受影响。
