# 开发任务书（TASK）

> 本文档定义 mah-score v0.8.0 的候选开发任务。
>
> v0.8.0 的目标不是新增计分规则，而是重构房间页的信息架构，让页面围绕两个核心任务展开：记录分数与查看本局账单。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.8.0**

主题：

房间页两段式结构重组。

目标：

让房间页从“房间信息 + 功能堆叠”调整为“记分 + 本局账单”的任务型页面。

核心判断：

- 记录分数是主要操作入口，通常只需要一个人操作。
- 本局账单是主要核对入口，四名玩家都需要能快速看懂。
- 房间号、邀请、玩家管理、历史、总账、撤销、结束游戏都是辅助功能，不应占据首屏主体。

核心目标：

- 弱化房间号和房间信息的视觉权重。
- 强化当前局记分入口。
- 默认展示四名玩家都能核对的本局账单。
- 减少页面中同时展开的信息量。
- 提高房间页的边界感和任务分区。
- 保持现有业务逻辑、API、Event Sourcing、数据模型不变。

要求：

- 保持 H5 优先。
- 保持 Mobile First。
- 不新增计分规则。
- 不新增统计能力。
- 不引入新的状态管理库。
- 不引入新的 UI 组件库。
- 不修改后端接口。
- 不复制 domain 业务逻辑到 UI。

# 四、问题定义

## 房间号不应作为页面主体

当前房间页顶部突出展示房间号，例如大号 `570`。

但房间号只是分享和识别用的信息，不是用户进入房间后的主要任务。

用户进入房间后最关心的是：

- 当前局能否继续记分
- 当前谁多少分
- 本局每个人收入、支出、净变化是多少
- 最近几笔计分是否正确

因此房间号应降级为薄状态条中的辅助信息。

## 页面内容太多且同时展开

当前页面会同时展示：

- 房间信息
- 三个统计卡片
- 玩家区
- 改名和删除操作
- 计分按钮
- 本局事件
- 我的本局账单
- 全员账单入口
- 历史局账单
- 玩家总账
- 邀请二维码
- 撤销和结束游戏

这些内容同时出现会削弱边界感，让用户不知道当前应该看哪里。

## 本局账单不应以“我的账单”为主

本局账单需要四名玩家一起核对。

默认只突出“我的本局账单”会让其他玩家难以快速确认自己的收支。

本局账单应默认展示所有玩家的本局收入、支出、净变化；当前玩家可以高亮，但不应隐藏其他玩家。

# 五、目标信息架构

房间页采用两段式主结构：

```text
[薄状态条]
房间 570 · 第 1 局 · 游戏中 · 已同步

[记分]
玩家当前总分
点炮 / 自摸 / 直杠 / 补杠 / 暗杠 / 流局
必要时显示番数选择、确认本局账单

[本局账单]
四名玩家本局收入、支出、净变化
本局事件明细

[更多]
玩家总账
历史局账单
邀请加入
撤销上一条
结束游戏
玩家管理
```

说明：

- 薄状态条不是一个大卡片。
- 记分区是第一主区域。
- 本局账单是第二主区域。
- 更多区域默认折叠或弱化。
- 不使用三段式常驻大区块，因为房间信息不值得成为独立主段。

# Phase 1：房间页主结构重组

## Epic 1

建立“记分 + 本局账单”的两段式页面结构。

### 目标

移除当前房间页顶部大房间号和统计卡片主视觉，让首屏直接服务记分和本局核对。

### 任务

调整 `RoomPage` 渲染结构：

- 顶部改为薄状态条。
- 删除独立三张统计卡片。
- 第一主区域为记分区。
- 第二主区域为本局账单区。
- 邀请、总账、历史、结束游戏、撤销放入次级区域。

### 涉及模块

`frontend/src/pages/RoomPage.tsx`

`frontend/src/styles.css`

### 验收标准

移动端首屏可以直接看到记分相关内容。

房间号不再作为页面最大视觉元素。

页面主结构能清楚分辨为记分区和账单区。

现有创建、加入、开始游戏、计分、撤销、同步行为不变。

`npm run check` 通过。

---

# Phase 2：本局账单全员默认展示

## Epic 2

将本局账单改为默认展示全员收支。

### 目标

让四名玩家都能直接核对本局收入、支出、净变化。

### 任务

调整本局账单展示：

- 默认展示所有玩家的本局账单。
- 每名玩家展示收入、支出、净变化。
- 当前玩家可以通过轻量样式高亮。
- 本局事件明细放在本局账单下方，作为核对依据。
- 移除“我的本局账单”作为主模块的展示方式。

### 涉及模块

`frontend/src/pages/RoomPage.tsx`

### 验收标准

四名玩家不需要展开额外区域即可看到自己的本局净变化。

当前玩家被识别时有轻量高亮。

本局事件仍能撤销单条事件。

本局确认账单流程不变。

`npm run check` 通过。

---

# Phase 3：次级功能收纳

## Epic 3

收纳低频功能，减少页面默认信息量。

### 目标

让邀请、历史、总账、房间管理、结束游戏等低频功能不干扰记分和本局核对。

### 任务

调整低频区域：

- 玩家总账默认折叠。
- 历史局账单默认折叠。
- 邀请加入默认折叠。
- 房间等待阶段保留玩家管理，但弱化改名和删除。
- 游戏阶段撤销上一条和结束游戏不作为主按钮组抢占账单区视觉。

### 涉及模块

`frontend/src/pages/RoomPage.tsx`

### 验收标准

页面默认信息量明显减少。

用户仍能找到邀请、撤销、结束游戏、历史、总账功能。

危险操作仍有确认。

`npm run check` 通过。

---

# Phase 4：移动端视觉收敛

## Epic 4

收敛视觉样式，提高边界感。

### 目标

减少边框、卡片嵌套和重复白底块，让移动端页面更像一个高频工具，而不是功能列表。

### 任务

调整样式：

- 减少卡片套卡片。
- 降低非核心区域边框密度。
- 统一按钮高度和间距。
- 保持点击区域适合单手操作。
- 收敛文字层级，避免过多大标题。
- 避免装饰性视觉，不增加无关图片或动画。

### 涉及模块

`frontend/src/pages/RoomPage.tsx`

`frontend/src/styles.css`

### 验收标准

移动端截图中主视觉聚焦在记分和本局账单。

页面区块边界清楚但不碎。

按钮文字不换行、不溢出。

没有明显重叠或布局跳动。

`npm run check` 通过。
