# 开发任务书（TASK）

> 本文档定义 mah-score v0.9.0 的开发任务。
>
> v0.9.0 的目标不是新增计分能力，而是建立一套轻量、可维护的前端 UI 基础设施，让房间页后续迭代不再依赖零散手写样式。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.9.0**

主题：

UI 基础设施与房间页组件化。

目标：

采用 `shadcn/ui` 的本地组件组织方式，结合现有 TailwindCSS，引入 Lucide React 图标，建立一套适合 mah-score 的轻量 UI 基础组件。

核心目标：

- 保持现有 TailwindCSS。
- 引入 `lucide-react` 用于工具型图标。
- 采用 shadcn/ui 的思想：组件代码进入本仓库，可按项目需要维护。
- 不引入完整重型 UI 框架。
- 暂不引入 Framer Motion。
- 把 RoomPage 中重复按钮、区块、账单行、记录行收敛为可维护组件。
- 保持业务逻辑、API、Event Sourcing、数据模型不变。

要求：

- Mobile First。
- UI 服务于计分和对账。
- 组件简单、可读、少抽象。
- 不新增计分规则。
- 不新增后端能力。
- 不改变 Redis 数据模型。
- 不引入 Redux 或全局状态库。
- 不使用大面积动画。

---

# 二、技术方案

采用：

```text
TailwindCSS
+ shadcn/ui 风格的本地组件
+ lucide-react
+ CSS transition
```

暂不采用：

```text
Framer Motion
```

原因：

- 当前应用是高频计分工具，动效不是核心能力。
- Framer Motion 会增加依赖和交互复杂度。
- 折叠、按钮 active、轻量状态变化可以先用 CSS transition。
- 如后续确认有明确动效场景，再单独评估引入。

不引入 Ant Design、MUI、Chakra UI。

不引入 Framer Motion。

不做主题切换。

不做暗色模式。

不重写业务逻辑。

不调整后端 API。

不调整 Event 类型。

# Phase 2：RoomPage 基础替换

## Epic 2

替换 RoomPage 中的基础按钮和区块。

### 目标

减少 RoomPage 中重复的 Tailwind class，让主页面视觉更统一。

### 任务

- 将主要按钮替换为 `Button`。
- 将主要白底区块替换为 `Section`。
- 将折叠区替换为 `Disclosure`。
- 保持交互行为不变。

### 验收标准

创建、加入、刷新、开始游戏、计分、确认下一局、撤销、结束游戏入口仍可用。

页面视觉不出现明显回退。

`npm run check` 通过。

低频功能仍可找到。
