# Phase 1：Event Sourcing 补齐

## Epic 1

Event Sourcing 补齐第一阶段

### 目标

把房间生命周期的核心动作改为追加 Event，减少 Room Hash 中的重复业务状态。

### 任务

以下操作必须追加 Event：

- ROOM_CREATED
- PLAYER_JOINED
- PLAYER_RENAMED
- PLAYER_REMOVED
- GAME_STARTED

Room Hash 仅保留：

- version
- status
- createdAt
- updatedAt
- 必要缓存

### 涉及模块

backend/services/roomService.ts

backend/services/eventStore.ts

shared/src/domain/replay.ts

frontend/src/pages/RoomPage.tsx

### API

POST /api/room/create

POST /api/room/join

POST /api/room/start

POST /api/room/player/rename

POST /api/room/player/remove

### Redis

Room Hash 继续保留最小状态缓存。

Event List 继续作为事实来源。

新增动作必须追加到 `room:{roomId}:events`。

### 风险

旧逻辑依赖 Room Hash 里的冗余字段。

前端读取状态的路径可能需要同时兼容旧数据。

### 验收标准

不再需要前端伪造 PLAYER_JOINED Event。

Replay 可恢复玩家列表、房间状态和分数。

旧房间数据兼容或有明确迁移策略。

---

## Epic 2

Replay 与旧数据兼容

### 目标

让 Replay 能完整恢复新事件模型下的房间状态，并兼容旧房间数据。

### 任务

支持：

- 旧事件继续 Replay
- 新事件继续 Replay
- RoomPage 读取 Replay 后的完整状态
- 旧房间数据有明确兼容策略

### 涉及模块

shared/src/domain/replay.ts

shared/src/types

backend/services/eventStore.ts

frontend/src/pages/RoomPage.tsx

### API

GET /api/room/{id}

GET /api/room/{id}/events

GET /api/room/{id}/sync?version=xx

### Redis

只读取现有事件和房间缓存，不引入新存储结构。

### 风险

兼容层过厚会让 Replay 逻辑变复杂。

旧数据和新数据在字段缺失时可能出现状态不一致。

### 验收标准

旧房间可正常打开。

新事件与旧事件均可 Replay。

房间页不依赖前端伪造状态。

# Phase 2：并发安全

## Epic 3

并发安全第一阶段

### 目标

降低多人同时加入房间、同时开始游戏时的数据覆盖风险。

### 任务

重点处理：

- 同时加入房间
- 同时开始游戏

方案优先级：

1. Redis 原子操作
2. Redis 事务
3. Lua

### 涉及模块

backend/services/roomService.ts

backend/services/eventStore.ts

backend/api/room/*.ts

### API

POST /api/room/join

POST /api/room/start

### Redis

加入和开始流程需要保证版本递增和状态写入的原子性。

### 风险

并发写入可能导致重复版本号。

并发加入可能导致玩家丢失。

### 验收标准

不会因为并发加入丢失玩家。

不会出现重复版本号。

不会出现同时开始覆盖状态的问题。

---

## Epic 4

并发安全第二阶段

### 目标

降低同时计分、同时撤销时的数据冲突风险。

### 任务

重点处理：

- 同时计分
- 同时撤销

### 涉及模块

backend/services/eventStore.ts

backend/api/room/*.ts

backend/services/roomService.ts

shared/src/domain/replay.ts

### API

POST /api/room/score

POST /api/room/undo

### Redis

计分和撤销事件必须保证追加顺序正确。

同一个目标事件不能被重复撤销。

### 风险

撤销与计分并发可能造成事件序列分叉。

重复请求可能制造重复业务结果。

### 验收标准

不会撤销同一个事件两次。

不会出现重复计分写入。

冲突请求返回明确错误。

# Phase 3：可观察性

## Epic 5

错误提示优化

### 目标

让非法请求、并发冲突、状态冲突都返回明确错误，便于定位和修复。

### 任务

统一失败返回：

- 明确 `code`
- 明确 `message`
- 不混用多套错误格式

重点覆盖：

- 房间不存在
- 房间状态不允许
- 版本冲突
- 重复操作
- 非法玩家参数

### 涉及模块

backend/api/room/*.ts

backend/services/*.ts

shared/src/types

### API

所有房间相关 API

### Redis

不新增存储结构。

只改善错误输出和校验路径。

### 风险

错误码设计过多会增加维护负担。

### 验收标准

所有失败场景都有可读错误信息。

前端可以直接展示错误原因。

---

## Epic 6

验收工具优化

### 目标

把 Replay、旧数据兼容、并发问题的验证变成可重复执行的工具。

### 任务

补齐或增强：

- Replay 检查脚本
- 并发写入验证脚本
- 旧数据兼容验证脚本
- 必要的回归检查命令

### 涉及模块

shared/scripts

backend/scripts

package.json

### API

无新增 API。

### Redis

仅用于本地或测试环境验证。

### 风险

工具做得过重会增加维护成本。

### 验收标准

可以重复执行验证关键回归点。

可以快速发现 Replay 或并发问题。
