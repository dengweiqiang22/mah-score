# 开发任务书（TASK）

> 本文档定义 mah-score v0.17.0 的候选开发任务。
>
> v0.17.0 的目标是收口流局退税解释能力，并降低 `RoomPage.tsx` 的继续维护成本。
>
> 如本文档与 AGENTS.md、ARCHITECTURE.md、DECISIONS.md 存在冲突，以后三者为准。

---

# 一、当前版本

**版本：v0.17.0**

主题：

流局事件明细解释优化与房间页职责拆分。

背景：

v0.16.0 / v0.16.1 已补齐流局结算、退税、流局录入 UI，并修正了本局结算的退税展示口径。

当前剩余两个问题：

- 流局退税后，原始杠牌事件仍在事件明细中展示，用户可能不确定这笔杠牌是否已经被退回；
- `RoomPage.tsx` 继续承载录入状态、流局状态、结算展示、历史记录、房主权限等逻辑，后续维护成本开始上升。

---

# 二、目标

本次目标：

- 让事件明细清楚表达“这笔杠牌已在流局时退税”；
- 保持事件流水完整，不隐藏、不删除、不改历史 Event；
- 将流局录入状态和当前局结算展示口径从 `RoomPage.tsx` 中拆出；
- 不引入新状态管理库；
- 不改变现有 API、Event payload 和 Replay 计分结果。

---

# 三、流局退税事件明细优化

## 问题

当前流局后：

- 本局结算展示的是退税后的有效账单；
- 流局记录下方会展示退税说明；
- 但原始杠牌记录仍像普通有效记录一样展示。

这会让用户产生疑问：

```text
这笔暗杠/补杠到底算不算？
为什么本局结算里没有这笔支出？
```

## 方案

在事件明细中，对已被流局退税覆盖的杠牌事件增加弱提示。

候选展示：

```text
张三 · 暗杠
已退税
```

或：

```text
张三 · 暗杠 · 已退税
```

要求：

- 标签使用低权重样式，不抢占主信息；
- 不隐藏原始杠牌事件；
- 不影响撤销入口；
- 不改变该事件原本的 `flows`；
- 只在本局存在有效 `DRAW_GAME`，且该杠牌玩家属于退税玩家时展示；
- 已撤销的杠牌事件不需要展示退税标签；
- 已撤销的流局事件不应让杠牌展示退税标签。

# 四、RoomPage 职责拆分

## 问题

`RoomPage.tsx` 当前职责偏多：

- 房间同步与 replay；
- 当前玩家与房主权限；
- 高频事件录入状态；
- 流局状态标记；
- 本局结算展示口径；
- 历史记录与撤销；
- 房间管理入口。

短期仍可维护，但继续叠加功能会增加理解成本。

## 拆分原则

只做低风险拆分，不做大重构。

优先拆出：

1. 流局录入状态
2. 当前局结算展示口径

暂不拆：

- API 调用；
- 页面整体布局；
- 房主权限；
- 历史面板结构；
- 房间管理弹层。

## 候选模块

```text
frontend/src/hooks/useDrawGameEntry.ts
```

负责：

- `drawGameStep`
- `drawFlowerPigPlayerIds`
- `drawNotReadyPlayerIds`
- `drawReadyFans`
- 玩家状态切换
- 流局 payload 生成辅助
- 流局状态摘要

```text
frontend/src/utils/roundSettlementDisplay.ts
```

负责：

- 从当前局 `ScoreHistoryItem[]` 中生成用于“本局结算”的有效事件列表；
- 识别已退税杠牌事件；
- 为事件明细提供 `已退税` 标记判断。

要求：

- 纯函数优先；
- 不依赖 React；
- 不依赖浏览器 API；
- 不复制 replay 计分逻辑；
- 命名清楚，避免过度抽象。

# 六、验收标准

- 流局退税后的杠牌事件在事件明细中出现 `已退税` 弱标签；
- 未退税杠牌不显示该标签；
- 撤销流局后，相关杠牌不再显示 `已退税`；
- 已撤销杠牌不显示退税标签；
- 本局结算继续使用退税后的有效账单；
- 最近记录、事件明细、本局结算三处口径一致；
- `RoomPage.tsx` 中流局录入相关状态明显减少；
- 当前局结算展示口径逻辑从 `RoomPage.tsx` 移出；
- `npm run check` 通过；
- `npm run check:replay` 通过；
- `npm run build` 通过。

---

# 七、风险控制

- 不改 Event Sourcing 模型；
- 不改历史 Event；
- 不新增依赖；
- 不把 UI 展示口径混入 replay 计分；
- 拆分后先保持行为一致，再做标签展示；
- 每次只移动一类逻辑，避免一次性大改。

---

# 八、非目标

本任务不处理：

- 房间规则配置页；
- 自摸加番/加底配置；
- 新的麻将牌型；
- 生产数据迁移；
- 大规模页面信息架构重做；
- 全局状态管理引入。
