# 体验迭代：房间创建与 Event 同步延迟基线

日期：2026-07-13

线上地址：https://mah-score.vercel.app/

本文件记录本次体验优化迭代开始前的线上实测结果、测试过程、代码观察和初步优化方向。后续迭代应以本文数据作为基线，优化后继续补充对比数据。

## 一、背景

当前线上版本部署在 Vercel，用户主要场景为中国人在印尼工业园区内使用。

已观察到的问题：

- 创建房间到进入房间可用状态，体感需要约 2 到 3 秒。
- Event 同步接近 1 秒。
- 每次同步都会触发页面组件刷新，影响使用体验。

本次目标不是立即修改代码，而是先完成线上实测，确认瓶颈位置，并形成可执行的优化迭代依据。

## 四、代码观察

### 1. 创建成功后整页跳转

位置：`frontend/src/pages/HomePage.tsx`

当前创建成功后调用：

```ts
window.location.assign(`/room/${response.data.roomId}`);
```

影响：

- 创建房间成功后会重新加载整个前端应用。
- 房间页重新初始化后还要请求 room 和 events。
- 这会把“创建房间”拆成“创建 API 等待 + 页面重载 + 房间数据 API 等待”。

### 2. 房间页首屏存在重复读取

位置：`frontend/src/pages/RoomPage.tsx`

当前 `loadRoom()` 并发请求：

```ts
const [roomResponse, eventsResponse] = await Promise.all([
  getRoom(roomId),
  getRoomEvents(roomId),
]);
```

后端 `getRoom()` 会读取 room hash 和全量 events，并通过 replay 得到 room 状态。

同时 `/api/room/events` 为了校验房间存在，也会先调用 `getRoom()`，然后再 `readRoomEvents(roomId)`。

影响：

- 房间页首屏至少存在多次 Redis event list 读取。
- room 和 events 在前端被拆成两个接口，但服务端为了计算 room 又依赖 events，导致重复读取。

### 3. sync 接口重复读取 event log

位置：`backend/api/room/sync.ts`

当前逻辑：

```ts
const room = await getRoom(roomId);

const data: SyncRoomEventsResponse = {
  events: await readRoomEventsAfterVersion(roomId, version),
  version: room.version,
};
```

其中：

- `getRoom(roomId)` 会读取全量 events 并 replay。
- `readRoomEventsAfterVersion(roomId, version)` 当前实现为先读取全量 events，再在内存中过滤版本。

影响：

- 即使没有新增 event，一次空同步也会读全量 event log。
- 当前实现还会对同一个房间 events 做重复读取。
- 房间打得越久，event log 越长，同步成本越高。

### 4. 空同步也触发整页重渲染

位置：`frontend/src/pages/RoomPage.tsx`

当前轮询逻辑每 3 秒执行一次，并设置：

```ts
setSyncStatus("syncing");
setSyncStatus("idle");
```

即使没有新事件，也会导致 `RoomPage` 重新执行。

`RoomPage` render 内包含多项派生计算：

- `replayRoomEvents`
- `createSettlement`
- `createScoreHistory`
- `createPlayerLedger`
- `createRoundLedgers`

影响：

- 空同步没有业务数据变化，却仍然触发整页 render。
- render 期间会重复执行 replay、结算、历史、账本等计算。
- 用户会感知到页面状态刷新或轻微抖动。

## 五、问题判断

当前体验问题不是单一原因导致，而是多个因素叠加：

1. Vercel Function 与 Redis 区域、用户所在地之间存在稳定网络往返。
2. API 内部对 event log 有重复读取。
3. 前端首屏把 room 和 events 拆成两个请求。
4. 创建房间后整页跳转，造成额外重载和二次请求。
5. 空同步也会更新 React state，触发整页派生计算。

因此，本次优化不应优先考虑 WebSocket 或新基础设施，而应先减少请求次数、减少 Redis 往返、减少无意义重渲染。

## 六、建议迭代方向

### Phase 1：减少重复读取和无效刷新

目标：

- 不改变 Event Sourcing 架构。
- 不引入新依赖。
- 不改变 Vercel + Vercel KV 技术路线。
- 优先降低创建后首屏和空同步体感延迟。

建议改动：

1. 新增或调整首屏读取接口，一次返回 `room + events`。
2. 房间页 `loadRoom()` 只请求一个接口。
3. `/api/room/sync` 不再调用 `getRoom()` replay 全量 events。
4. sync 接口只读取 room hash 的 version，并按版本增量读取 events。
5. 前端 sync 返回空 events 时，不更新会导致整页 render 的 UI 状态。
6. 有新增 events 时，前端直接追加 events，而不是调用 `loadRoom()` 全量刷新。

预期收益：

- 房间页首屏从两个请求降为一个请求。
- 空同步 Redis 读取从全量读取降为轻量读取。
- 空同步不再驱动整页业务派生计算。
- 用户体感上“同步中”的闪动减少。

### Phase 2：创建房间体验优化

目标：

- 创建成功后尽快进入可用房间页。
- 避免创建成功后马上重复读取同一批初始数据。

建议改动：

1. `/api/room/create` 返回初始 `room` 和 `events`。
2. 前端创建成功后把初始数据带入 RoomPage。
3. 避免 `window.location.assign()` 导致整页重载，可改为轻量客户端路由状态切换。

注意：

- 当前项目没有引入 React Router，不建议为了这一个点马上新增依赖。
- 可以先用现有 `App` 的 pathname 判断模式，做一个轻量的本地导航状态封装。

### Phase 3：派生计算 memo 化

目标：

- 降低 React render 成本。
- 避免非业务数据变更导致 replay、settlement、history、ledger 重算。

建议改动：

1. 对 `replayRoomEvents` 使用 `useMemo`。
2. 对 `createSettlement` 使用 `useMemo`。
3. 对 `createScoreHistory`、`createPlayerLedger`、`createRoundLedgers` 使用 `useMemo`。
4. 将同步状态展示拆成更小组件，避免影响整个 RoomPage。

### Phase 4：区域配置检查

目标：

- 在代码层优化后，再确认基础设施是否还有稳定收益。

建议检查：

1. Vercel Function 实际运行区域。
2. Vercel KV / Upstash Redis 实际区域。
3. Function 区域和 Redis 区域是否一致。
4. 对印尼用户来说，新加坡区域是否仍是最优。

说明：

- 区域优化可能降低 RTT。
- 但如果 API 继续重复读取和全量 replay，区域优化无法解决根本问题。

## 七、不建议本轮引入的方案

### 1. 暂不改 WebSocket

原因：

- 项目宪法明确当前同步机制采用 Polling。
- 当前问题主要来自重复请求、重复读取和无效重渲染。
- WebSocket 会增加连接状态、断线重连、部署和调试复杂度，不符合当前 MVP 阶段的维护成本目标。

### 2. 暂不引入全局状态管理库

原因：

- 当前状态规模仍可由本地 state、hook、`useMemo` 管理。
- 引入 Redux 或类似库会增加维护成本。
- 项目宪法要求优先 Local State，未经允许不引入 Redux。

### 3. 暂不更换后端或数据库

原因：

- 项目宪法明确后端统一为 Vercel，数据存储为 Vercel KV。
- 当前瓶颈可以在现有架构内先优化。

## 八、下一步验收指标

完成 Phase 1 后，建议重新测量：

| 指标 | 当前基线 | 目标 |
| --- | ---: | ---: |
| 房间首屏完整数据 | 约 0.95s | 降低到 0.5s 到 0.7s 区间，或至少减少一次请求 |
| 空同步 | 0.89s 到 0.93s | 明显低于当前基线，且不触发整页业务计算 |
| 创建后进入房间可用状态 | 约 4s 级别 | 明显减少二次加载等待 |
| 空同步页面刷新感 | 明显存在 | 基本不可感知 |

验收时应记录：

- 首页耗时
- 创建房间耗时
- 房间首屏接口耗时
- 空同步连续 5 次耗时
- 有新增 event 的同步耗时
- React 页面是否仍因空同步重算 replay 和 ledger

## 九、结论

本次实测确认：线上体验问题主要集中在创建房间和 event 同步链路。

最值得优先处理的是：

1. 减少 room/events/sync 的重复 Redis 读取。
2. 让 sync 真正按增量事件工作。
3. 避免空同步触发整页 React 重渲染。
4. 创建房间后复用初始数据，减少二次加载。

这些优化符合当前项目的 MVP 原则和 Event Sourcing 架构，不需要引入新基础设施，也不需要改变 Vercel + Vercel KV 的技术路线。
