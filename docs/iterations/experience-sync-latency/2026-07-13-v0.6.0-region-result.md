# 体验迭代 v0.6.0：指定 Vercel Function 区域后的复测结果

日期：2026-07-13

线上地址：https://mah-score.vercel.app/

对比文档：

- `docs/iterations/experience-sync-latency/2026-07-13-baseline.md`
- `docs/iterations/experience-sync-latency/2026-07-13-v0.6.0-result.md`

本文件记录在重新部署并指定 Vercel Function 区域后的线上复测结果。

## 一、背景

v0.6.0 代码优化后，线上表现已经在结构上改善：

- 房间页首屏请求数从两个减少为一个。
- sync 接口不再全量 replay。
- 前端空同步不再触发整页 state 更新。
- 创建房间响应包含初始 `room + events`。

但上一轮线上复测显示：

- 创建房间仍接近 3 秒。
- 加入房间仍接近 2.75 秒。
- 空同步连续 5 次均值约 0.92 秒。

后续确认主要原因是 Vercel Function 未显式指定区域，导致 Function 与 KV/Redis 往返链路较长。重新指定 Function 区域并部署后，本文件重新测量关键路径。

## 二、测试环境

测试端：

- Codex 工作区：`/home/dev/workspace/mah-score`
- 测试方式：`curl` 分段耗时统计、本地响应内容校验

线上环境：

- Vercel Production
- Vercel Functions
- Vercel KV / Upstash Redis
- 已重新部署并指定 Vercel Function 区域

本次测试创建的生产测试房间：

`813`

该房间仅用于本次区域配置后的优化验证。

## 五、结果判断

本次复测确认：线上主要瓶颈不仅是代码中的重复读取和无效刷新，也包括 Vercel Function 与 KV/Redis 的区域距离。

v0.6.0 代码优化解决了：

- 重复请求
- 重复读取
- 全量 replay
- 前端空同步刷新感
- 增量 events 应用

指定 Vercel Function 区域后，进一步解决了：

- Function 到 KV/Redis 往返过慢
- 创建房间和加入房间写操作接近 3 秒的问题
- 空同步稳定接近 1 秒的问题

当前结果已经进入基本可用区间：

- 创建房间约 0.7 秒
- 加入房间约 0.4 秒
- 房间详情约 0.3 秒
- 空同步中位数约 0.3 秒

## 六、后续建议

短期内不建议引入 WebSocket 或更换后端平台。

下一步可以继续做小步优化：

- 保持 Function 区域配置，避免回退。
- 将区域配置写入部署文档，防止后续遗漏。
- 后续如继续优化创建和加入，可关注 Redis 写入次数和锁释放脚本耗时。
- 用真实手机网络再做一次端到端主观体验验证。

本轮结论：

v0.6.0 代码优化 + Vercel Function 区域配置后，房间创建和同步体验已经达到 MVP 可用水平。
