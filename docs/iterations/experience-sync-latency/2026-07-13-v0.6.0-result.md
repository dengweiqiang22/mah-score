# 体验迭代 v0.6.0：房间创建与 Event 同步优化结果

日期：2026-07-13

线上地址：https://mah-score.vercel.app/

对比基线：`docs/iterations/experience-sync-latency/2026-07-13-baseline.md`

本文件记录 v0.6.0 部署后的线上复测结果。测试方式尽量沿用基线阶段的 `curl` 分段耗时统计。

## 一、本次上线内容

v0.6.0 已完成以下优化：

- 房间页首屏读取从 `/api/room` + `/api/room/events` 合并为单次 `/api/room`。
- `/api/room` 现在返回 `room + events`。
- `/api/room/sync` 空同步不再全量读取 events 和 replay。
- 前端同步返回新增 events 后直接追加，不再调用全量 `loadRoom()`。
- 空同步成功且无新增 events 时，不再更新会触发整页 render 的 React state。
- 房间页核心派生数据使用 `useMemo` 缓存。
- 创建房间响应返回初始 `room + events`，前端创建成功后通过 `sessionStorage` 一次性复用初始数据。
- 删除独立 `/api/room/detail` 函数，避免 Vercel Hobby 12 个 Serverless Functions 上限。

### 2. 创建房间

```bash

  -H 'Content-Type: application/json' \
  --data '{"nickname":"优化验证"}' \
  -o /tmp/mah_v060_create.json \
  -w 'dns=%{time_namelookup}\nconnect=%{time_connect}\ntls=%{time_appconnect}\nttfb=%{time_starttransfer}\ntotal=%{time_total}\nsize=%{size_download}\n' \
  https://mah-score.vercel.app/api/room/create
```

结果：

| 指标 | 耗时 |
| --- | ---: |
| DNS | 0.007674s |
| TCP Connect | 0.056720s |
| TLS | 0.141349s |
| TTFB | 2.983383s |
| Total | 2.983468s |
| Size | 770 bytes |

响应内容校验：

```json
{
  "roomId": "966",
  "roomVersion": 2,
  "eventCount": 2,
```

## 五、结果判断

本次 v0.6.0 优化在结构上是有效的：

- 房间页刷新首屏请求从两个接口减少为一个接口。
- 创建房间后前端可以复用创建响应中的初始数据，避免立即二次读取。
- sync 接口从全量读取和 replay 改为轻量版本读取与增量 event 读取。
- 前端可以增量应用新增 events。
- 空同步成功且无新增 events 时，不再触发整页业务派生计算。

但线上实测也说明：

- 创建房间、加入房间这类写操作仍接近 3 秒。
- 空同步连续测量均值与基线接近，平台和网络往返仍是主要耗时来源。
- 单次空同步和 v3 空同步能降到约 0.65 到 0.68 秒，说明轻量路径有收益，但并不稳定。

因此，本轮优化主要改善的是：

1. 请求数量
2. 重复读取
3. 前端刷新感
4. 增量同步正确性

而不是完全解决 Vercel Function 到 Redis 链路的网络延迟。

## 六、后续建议

下一轮如果继续优化体验，建议优先关注写操作链路：

- 创建房间当前会执行多次 Redis 写入和读取，仍接近 3 秒。
- 加入房间也接近 2.75 秒。
- 可以继续减少写操作后的回读，或用更少 Redis 往返完成创建和加入。

基础设施层面也应检查：

- Vercel Function 实际运行区域。
- Vercel KV / Upstash Redis 实际区域。
- Function 与 Redis 是否同区域。

如果两者不在同一区域，代码层优化很难彻底消除 0.7 到 1 秒级别的同步等待。
