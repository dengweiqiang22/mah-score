# mah-score 设计系统

> 本文档定义 mah-score 当前阶段的轻量视觉规范。
>
> 目标不是建立复杂设计系统，而是让 H5 房间页从“组件化原型”演进为统一、克制、适合牌桌单手操作的计分工具。

---

# 一、产品定位

mah-score 是一个轻量四川麻将计分工具。

主要使用场景：

- 手机竖屏；
- 牌桌旁；
- 单手操作；
- 快速记录；
- 多人通过分享链接加入；
- 未来可能接入微信小程序。

视觉关键词：

- 轻量；
- 稳定；
- 高效；
- 清晰；
- 温和；
- 有轻微麻将氛围。

不采用：

- 企业后台风格；
- 传统红金棋牌大厅风格；
- 赌场风格；
- 大量渐变、投影和装饰；
- 所有内容都包进同重量 Card 的结构。

---

# 二、颜色

当前阶段优先使用 TailwindCSS 内置色阶，不新增复杂主题系统。

## 页面背景

```text
bg-stone-100
```

说明：

- 整页使用暖灰白；
- 避免大面积纯白；
- 白色只用于需要聚焦的局部容器。

## 主色

```text
emerald-700
emerald-800
```

用于：

- 主按钮；
- 当前主体玩家；
- 成功反馈；
- 关键选中状态。

## 浅主色

```text
emerald-50
emerald-100
emerald-200
```

用于：

- 轻量选中背景；
- 当前用户标签；
- 状态提示；
- 低强度强调。

## 正负分

正分：

```text
text-emerald-700
```

负分：

```text
text-red-700
```

零分：

```text
text-stone-500
```

要求：

- 正负分必须保留 `+` 和 `-` 符号；
- 不能只依赖颜色表达输赢；
- 分数统一使用 `tabular-nums`。

## 警告与危险

警告：

```text
amber-50
amber-900
amber-200
```

危险：

```text
red-50
red-700
red-200
```

警告色只用于真实异常、确认风险和危险操作，不用于普通流程装饰。

## 正文

主要文字：

```text
text-stone-950
```

次要文字：

```text
text-stone-500
```

弱提示：

```text
text-stone-400
```

# 三、字号

页面标题：

```text
text-2xl font-semibold
```

区块标题：

```text
text-base 或 text-lg font-semibold
```

主要数据：

```text
text-xl 到 text-3xl font-semibold tabular-nums
```

正文：

```text
text-sm 到 text-base
```

辅助信息：

```text
text-xs 到 text-sm
```

原则：

- 工具界面内不使用 hero 级大字；
- 操作区域标题要紧凑；
- 分数比头像装饰更重要。

---

# 四、间距

页面水平边距：

```text
px-4
```

主区块间距：

```text
gap-3 到 gap-4
```

区块内部间距：

```text
p-3 到 p-4
```

紧凑列表：

```text
gap-2
py-2
```

原则：

- 首屏优先看见房间状态、玩家总分、事件工作台；
- 不用大面积留白制造高级感；
- 高频控件之间保持稳定距离，便于形成操作记忆。

---

# 五、圆角与阴影

圆角：

```text
rounded-md
```

胶囊标签：

```text
rounded-full
```

阴影：

```text
shadow-sm
```

原则：

- 当前阶段统一使用中等圆角；
- 不使用大圆角卡片；
- 不让每个模块都有明显投影；
- 优先通过背景、留白、分隔线和字号区分层级。

# 六、组件规范

## Button

按钮用于明确动作。

主按钮：

```text
bg-emerald-700 text-white
```

次按钮：

```text
bg-stone-100 text-stone-900
```

危险按钮：

```text
bg-red-50 text-red-700
```

要求：

- 高频主按钮高度不低于 `h-12`；
- 工具按钮可使用 Lucide 图标；
- 图标只辅助识别，不替代文字；
- 不给每个按钮都机械添加图标。

## 标签

标签用于极少量关键状态：

- 我；
- 已同步；
- 同步失败；
- 已撤销；
- 最高分。

要求：

- 标签体积小；
- 不喧宾夺主；
- 不把当前用户整块染绿，避免和选中/正分语义冲突。

## 分数

统一使用产品组件 `ScoreValue`。

要求：

- 正分显示 `+4`；
- 负分显示 `-6`；
- 零分显示 `0`；
- 使用 `tabular-nums`；
- 根据语义选择尺寸。

# 七、房间页模式

## 高频记录页

视觉任务：

- 快速看分；
- 快速记一笔；
- 快速撤销最近错误。

结构优先级：

```text
顶部房间状态
玩家总分区
状态提示
固定工作台
最近记录
低频入口
```

不应继续表现为多个同重量 Card 纵向堆叠。

## 本局结算页

视觉任务：

- 快速读懂输赢；
- 确认下一局；
- 必要时查看明细。

结构优先级：

```text
顶部房间状态
玩家总分区
本局结果
本局净变化
确认下一局
事件明细
```

## 入口页

视觉任务：

- 快速创建房间；
- 快速加入房间。

要求：

- 不做营销页；
- 不做巨大 hero；
- 表单和主按钮集中；
- mah-score 作为产品名出现，但不要像开发项目代号。

---

# 八、玩家 Tile 状态

玩家组件必须能表达：

- 默认；
- 当前用户；
- 主体玩家；
- 责任玩家；
- 已胡牌禁用；
- 普通禁用；
- 长昵称截断。

主体玩家：

- 胡牌者；
- 自摸玩家；
- 杠牌玩家。

视觉建议：

```text
bg-emerald-50
ring-emerald-200
border-emerald-600
```

责任玩家：

- 点炮者；
- 引杠玩家。

视觉建议：

```text
bg-red-50
ring-red-100
border-red-400
```

当前用户：

- 使用轻量“我”标签；
- 不改变整块主色。

---

# 九、事件按钮状态

固定事件：

- 直杠；
- 补杠；
- 暗杠；
- 点炮；
- 自摸；
- 流局。

每个事件按钮必须支持：

- 默认；
- selected；
- disabled；
- active；
- focus-visible。

要求：

- 事件区是高频记录页视觉核心；
- 事件按钮高度适合单手点击；
- 不使用六种不同颜色；
- Lucide 图标只作为辅助，文字必须保留。

---

# 十、空状态

空状态应轻量。

例如：

```text
尚无本局记录
完成一次记分后会显示在这里
```

要求：

- 不使用大块灰色占位；
- 不让空状态占据主流程；
- 空状态文案短。

---

# 十一、响应式规则

主要验收尺寸：

```text
375 × 812
390 × 844
430 × 932
768 × 1024
```

检查内容：

- 3 名玩家；
- 4 名玩家；
- 长昵称；
- 两位和三位数分数；
- 当前用户为正分；
- 当前用户为负分；
- 主体玩家和责任玩家；
- disabled 状态；
- 本局结算状态。

要求：

- 不出现文字溢出；
- 不出现按钮挤压；
- 不出现信息重叠；
- 普通记分路径尽量不滚动；
- 玩家和事件能快速识别；
- 低频功能不抢主流程。

---

# 十二、截图驱动迭代

UI 任务不能只看代码验收。

每轮 UI 实现后必须：

1. 启动本地页面；
2. 使用移动端视口截图；
3. 检查是否仍像后台；
4. 检查是否有过多 Card；
5. 检查玩家和事件是否清楚；
6. 检查普通记分是否需要滚动；
7. 根据截图做一次二次优化。

截图发现的问题应写入阶段汇报。
